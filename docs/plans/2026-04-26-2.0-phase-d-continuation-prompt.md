## Things that bit C1–C5 — watch for them

1. **Auto-seeded "Future" release** (`type=release`, `status=planning`, `priority=4`) is in every freshly initialized dashboard test DB. List endpoints over the issues collection see it. Either filter it out in seeded tests or write tests that tolerate its presence.
2. **mypy doesn't narrow `str` against tuple membership.** `if raw not in ("slim", "full"): ... else: return raw  # mypy: still str, not Literal["slim","full"]`. Use explicit `if raw == "slim": return "slim"` branches when you need Literal narrowing. (C5 hit this; the workaround is in `dashboard_routes/common.py::_parse_response_detail`.)
3. **PT018 lint splits composite asserts.** `ruff` will reject `assert isinstance(x, list) and len(x) == 1`. Write them as two separate assert lines from the start.
4. **TypedDict inheritance can't drop keys.** If you find yourself wanting to "rename `SlimIssue.id` to `SlimIssue.issue_id` while keeping `id` available on extending types," that doesn't work — the inherited key stays. The MCP D3 rename works because nothing extends `SlimIssue` other than `BlockedIssue`, which is also fine with `issue_id`.
5. **`db.get_comments` returns `[]` for unknown issue ids — does not raise.** D2's `get_comments` test for "unknown issue" cannot rely on a `KeyError` 404; it returns an empty list. The MCP wrapper inherits this behavior. Decide if that's the contract you want or if MCP should validate up front (pattern from C4).
6. **`_check_id_prefix` only fires on cross-project ids.** Same-prefix nonexistent ids hit `KeyError` in handler bodies, not at the prefix check. For 404 fixtures, use a same-prefix id like `genp-deadbeef00`.
