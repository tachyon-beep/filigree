### 1. Cross-surface parity regression test module — DO THIS FIRST

**Why first:** the bed-down found cross-surface divergences
after-the-fact; a regression module would have caught them all at once
and will prevent future drift as Stage 2B rewrites batch/list shapes.

**What:** a new test class that fires the same bad input at dashboard
(`AsyncClient`), MCP (`call_tool`), and CLI (`CliRunner --json`) and
asserts **identical** envelope (`error`, `code`, and — where applicable
— HTTP status / CLI exit code parity).

**Suggested cases (drawn from what the bed-down touched):**
- Blank/whitespace `actor` on an update.
- Blank/whitespace `assignee` on claim/claim_next.
- Out-of-range priority on create, update, batch_update.
- Unknown issue type on create.
- Invalid status transition on update.
- Already-closed issue on close.
- Unknown `issue_id` on get/update/close.

**Where:** `tests/util/test_cross_surface_parity.py` (new). Use the
existing fixtures: `client` (dashboard, from
`tests/api/conftest.py`), `mcp_db` + `call_tool` (from
`tests/mcp/conftest.py`), `cli_in_project` (from `tests/cli/conftest.py`).

### 2. Rebaseline the Stage 2B plan

**Why:** Stage 2a execution was rough enough that the original 2B plan
is not trusted. Scope has shifted materially since 2a started.

**Concrete changes to write into a new
`docs/plans/2026-04-23-2.0-stage-2b-rebaseline.md` (or similar):**

- **2b.0 scope shrinks** to type-alias retirement only.
  `BatchFailureDetail.code` is already `ErrorCode` across every surface
  (landed in commits `11cfb80` + `dc3917e`). What remains: migrate call
  sites from `BatchFailureDetail` TypedDict to the generic
  `BatchFailure[_T]` / `BatchResponse[_T]` envelopes defined in
  `src/filigree/types/api.py`. This is a rename, not a wire-contract
  change.
- **Document the `classify_value_error` boundary rule.** State-machine
  sites (update/close/reopen, batch_close/batch_update) use it;
  input-validation sites (`_safe_path`, URL parsing, schema validation
  in `mcp_tools/files.py` / `mcp_tools/scanners.py` / `mcp_tools/meta.py`
  / `mcp_tools/observations.py`) must stay hardcoded `VALIDATION`.
  Misapplying the heuristic to input-validation paths would regress
  cases the bed-down already got right.
- **Clarion-facing `POST /api/v1/scan-results`** remains the highest-risk
  hop in 2B (flagged W12:1 in the plan review). Gate its commit with a
  Clarion dogfood run *before* landing the shape change.
- **Revert readiness** stays per-commit (good pattern from the original
  plan), but the revert target must be a clean commit — don't layer 2B
  on top of an unstable base.
- **Carry forward architectural deferral:** CLI `claim` and MCP
  `_handle_claim_issue` still hardcode `CONFLICT` for the
  post-pre-validation `ValueError` path. Reachable messages:
  - "Cannot claim {id}: already assigned to '{other}'" — CONFLICT is
    correct.
  - "Cannot claim {id}: status is '{x}', expected open-category state"
    — arguably `INVALID_TRANSITION`, currently `CONFLICT`. Both surfaces
    are *consistent*, so a client branching on code behaves the same
    way. Retiring this heuristic cleanly requires typed exceptions from
    `claim_issue` (`AlreadyAssignedError`, `WrongStatusError`) — Stage 3
    scope.
