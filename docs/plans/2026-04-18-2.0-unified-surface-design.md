# Filigree 2.0 — Unified Surface Design

**Date:** 2026-04-18
**Author:** Claude (Opus 4.7) with John Morrissey
**Branch:** `2.0-project-management-extension`
**Status:** Design — approved for planning
**Source review:** [`2026-04-18-mcp-cli-interface-review.md`](./2026-04-18-mcp-cli-interface-review.md)
**Review feedback incorporated:** senior-user review pass, 2026-04-18.

## Summary

Filigree 2.0 aligns the MCP and CLI surfaces into one coherent contract.
Seven recommendation families from the first-principles review ship as a
bundled breaking release, plus concrete follow-throughs from the senior-user
review:

1. Unified `BatchResponse` and `ListResponse` envelopes plus a closed
   `ErrorCode` enum (`src/filigree/types/api.py`).
2. `id` → `issue_id` everywhere in MCP tools, CLI output, and response JSON.
3. `state` → `status` everywhere user-facing; "state" retires as an API word.
4. New composed operations `start_work` / `start_next_work` (atomic claim
   + transition, with explicit `target_status` override).
5. Full CLI parity for observations, files/findings, and scanners — 19 new
   commands.
6. `get_issue.include_files` defaults to `False`.
7. Schema-version-mismatch UX routed through a typed exception, plus a
   forward-looking warning from `filigree init` when the installed tool is
   older than the schema it just created.

The bundle is deliberate: 2.0 concentrates all the breakage in one release.

## Context

### Why this release, why now

Two pressures converged:

- **Review evidence** — agents observably stumble over the surface (see the
  source review for documented pain points: multi-named IDs, half-composed
  `claim`, drifting batch-response shapes, three words for the same concept,
  CLI capability holes). The pain is real and reproducible with the current
  tools.
- **2.0 is the right place** — the branch is already the release staging
  area; breaking changes belong in the semver-major cut, not sprinkled
  across minors.

A possible future pressure also exists: filigree *may* come to be consumed by
sibling tools (Clarion, Wardline — see the Loom doctrine at
`~/clarion/docs/suite/loom.md`). The current design accommodates that
direction without requiring it. Every change in this release is defensible
on single-product grounds; the cross-product framing is a secondary
justification, not the premise. See §Cross-product considerations for the
narrow set of contracts that matter if siblings materialize.

## Non-goals

- **No auto-migration / downgrade.** Schema migrations remain forward-only.
- **No capability negotiation.** Any cross-product version skew is handled
  bilaterally at release time, not by in-protocol handshake.
- **No dual-accept compat shim.** 2.0 is a clean cut; no `id`/`issue_id`
  both-accepted phase.
- **No new sibling product APIs.** This release aligns the existing surface;
  protocol additions are out of scope.
- **No general refactoring** beyond what these changes require.

## Design

### 1. Unified contract — `BatchResponse`, `ListResponse`, `ErrorCode`

**Location:** extend `src/filigree/types/api.py` (current 387 lines; already
hosts `SlimIssue`, `ErrorResponse`, `TransitionDetail`).

#### 1a. Batch response envelope

```python
class BatchResponse(TypedDict, Generic[T]):
    succeeded: list[T]           # T defaults to SlimIssue for issue ops;
                                 # full record type for finding/observation ops.
                                 # See §1d for the rule.
    failed: list[BatchFailure]   # one entry per failed id; always present.
    newly_unblocked: NotRequired[list[SlimIssue]]  # only when the op can
                                                    # unblock (close/transition)

class BatchFailure(TypedDict):
    item_id: str        # id of the failed item — issue_id, finding_id, or
                        # observation_id depending on which batch tool.
    error: str          # human-readable message
    code: ErrorCode
```

Rules:

- `failed` is always present; empty list when no failures.
- `newly_unblocked` is omitted entirely when meaningless (not set to empty
  list as noise). Present only for close/transition-style ops that can
  unblock.
- Dead keys removed: `closed`, `errors`, `results`, `count`, `updated`.
  `batch_add_label` loses its duplicate `succeeded`+`results` pair.
- **Empty-input semantics.** A batch call with an empty `ids` list returns
  `{"succeeded": [], "failed": []}` (a valid-but-trivial response) — not an
  error. Agents constructing batches dynamically shouldn't have to guard
  against "what if the filter matched nothing."
- **Malformed-input semantics.** If `ids` is missing entirely, or elements
  are the wrong type, the call returns an `ErrorResponse` with
  `code: VALIDATION` — not an empty batch response. Schema failures are
  not batch failures.

#### 1b. Response detail opt-in for batch operations

Every batch tool accepts an optional `response_detail` parameter:

```python
# MCP: batch_close, batch_update, batch_add_label, batch_add_comment,
#      batch_dismiss_observations, batch_update_findings
response_detail: Literal["slim", "full"] = "slim"
```

- `"slim"` (default): `succeeded: list[SlimIssue]` for issue ops, compact
  records for others. Cheap.
- `"full"`: `succeeded` contains full records — full `IssueDict` for issue
  ops, full finding/observation records otherwise. Use when the caller
  needs to verify custom fields landed, confirm a status transition, etc.,
  without re-fetching.

CLI mirror: `--detail=slim|full`.

This closes the ergonomic regression the senior-user review flagged
("after `batch_update --status=fixing`, I want to verify custom fields
landed correctly"). Slim stays the default — cheap — and full is available
by opting in.

#### 1c. List response envelope

List reads also get a standardized envelope. The current cap
(`_MAX_LIST_RESULTS = 50` in `mcp_tools/common.py`) is inconsistently
surfaced — `list_observations` returns `has_more`, others don't — and the
shape drift matches what `BatchResponse` fixes for mutations.

```python
class ListResponse(TypedDict, Generic[T]):
    items: list[T]
    has_more: bool
    next_offset: NotRequired[int]  # present when has_more=True, absent otherwise
```

Applied to: `list_issues`, `search_issues`, `list_labels`, `list_files`,
`list_findings`, `list_observations`, `list_scanners`, `list_packs`,
`list_types`, `get_ready`, `get_blocked`, `get_comments`,
`get_issue_events`, `get_issue_files`, `get_changes`.

- `items` is always present; empty list when no results.
- `has_more` is always present (never omitted) so callers can't mistake
  absence for "no more."
- `next_offset` is present when the next page exists; callers pass it back
  via `offset` on the subsequent call.

#### 1d. Type parameterization rule

`BatchResponse` and `ListResponse` are generic in `T`. The project uses two
`T` binds:

- **Issue-centric ops** → `BatchResponse[SlimIssue]` (default) or
  `BatchResponse[IssueDict]` (when `response_detail="full"`).
- **Non-issue ops** (findings, observations, files) → `BatchResponse[FindingRecord]`,
  `BatchResponse[ObservationRecord]`, `BatchResponse[FileRecord]` (slim) or
  full-record variants when opted in.

Type aliases named in `api.py` make this explicit; tests parametrize over
the concrete variants rather than treating all batch responses as one
shape.

#### 1e. Error code enum

Replace 15 ad-hoc strings with a closed 9-member set:

```python
class ErrorCode(StrEnum):
    VALIDATION          # was: invalid, validation_error, invalid_path
    NOT_FOUND           # was: not_found, scanner_not_found,
                        #      command_not_found, unknown_tool
    CONFLICT            # claim race, optimistic-lock miss
    INVALID_TRANSITION  # unchanged
    PERMISSION          # was: permission_error
    NOT_INITIALIZED     # unchanged
    IO                  # was: io_error, db_error (see note below)
    INVALID_API_URL     # unchanged
    STOP_FAILED         # unchanged
```

Consolidation rationale:

- `invalid`/`validation_error` — same meaning across different files.
- `io_error`/`db_error` → `IO`. Callers do not currently branch on which;
  when retry semantics matter in the future, a separate `retryable: bool`
  field on the error response is the right addition — not another code.
  (Recorded in Open Questions as a future consideration.)
- `not_found`/`scanner_not_found`/`unknown_tool`/`command_not_found` — all
  "you named a thing that doesn't exist"; disambiguation belongs in the
  `error` message.

#### 1f. `ErrorResponse` shape alignment and wire example

MCP currently uses flat `{"error": str, "code": str}`; dashboard routes use
nested `{"error": {"message", "code", "details"}}`. 2.0 standardizes on the
flat MCP shape with an optional `details: dict[str, Any]` for rare
structured data:

```python
class ErrorResponse(TypedDict):
    error: str                           # human-readable message
    code: ErrorCode                      # the closed enum
    details: NotRequired[dict[str, Any]] # structured specifics; optional
```

Wire example:

```json
{
  "error": "cannot claim issue filigree-abc123: already assigned to alice",
  "code": "CONFLICT",
  "details": {
    "issue_id": "filigree-abc123",
    "current_assignee": "alice"
  }
}
```

Dashboard's `_error_response()` in `dashboard_routes/common.py` rewrites to
emit this flat form.

### 2. Naming normalization — `issue_id` and `status`

**Rule: the issue ID is always `issue_id`.**

Renamed parameters (MCP tool `inputSchema` and TypedDicts in
`types/inputs.py`):

| Tool | Before | After |
|---|---|---|
| `get_issue`, `update_issue`, `close_issue`, `reopen_issue`, `claim_issue`, `release_claim`, `undo_last` | `id` | `issue_id` |
| `add_dependency`, `remove_dependency` | `from_id`, `to_id` | `from_issue_id`, `to_issue_id` |
| `create_issue` | `parent_id` | `parent_issue_id` |

Response JSON: `SlimIssue.id` renamed to `SlimIssue.issue_id`. Asymmetry
between input (`issue_id`) and output (`id`) is exactly the friction the
review flagged.

Exceptions that stay unchanged:

- `milestone_id` — different entity.
- `status_category`, `StatusCategory` enum — these *are* categories; naming
  is correct.
- `BatchFailure.item_id` — deliberately generic; see §1a.

**Rule: user-facing "state" becomes "status".**

| Tool / CLI | Before | After |
|---|---|---|
| MCP | `get_workflow_states`, `explain_state` | `get_workflow_statuses`, `explain_status` |
| CLI | `workflow-states`, `explain-state` | `workflow-statuses`, `explain-status` |
| TypedDict field on transition | `category` (3-valued roll-up) | unchanged — distinct from `status` |

Internal type names (`WorkflowState`, etc.) stay; they are not user-facing.
The DB column is already `status`; no migration.

**Doc updates.** CLAUDE.md (root + project) replaces the `filigree update
<id> --status=in_progress` "claim it" line with `filigree start-work
<issue-id> --assignee <name>`. AGENTS.md mirrors.

**CLI positional-argument order: `batch-add-label` and `add-label`.**

Current: `add-label <id> <label>` (id first) vs `batch-add-label <label>
<ids...>` (label first; variadic forces it to come last). Inconsistent and
reviewer-flagged as a future scar if deferred.

**Fix in 2.0:** align both on `<label> <ids...>` shape. `add-label <label>
<id>` accepts a single id in the variadic position; `batch-add-label <label>
<ids...>` unchanged. Same verb, same positional order, same mental model.
Lands in Stage 6 with the other renames.

### 3. Composed operations — `start_work` and `start_next_work`

**Design intent.** `claim_issue` is the right atomic primitive (optimistic
CAS on `assignee`). It is also only half a workflow — an agent next needs
to `update_issue --status=in_progress`, and that call has no atomicity on
assignee. Multiple producers racing on the same ready issue produce real
divergence.

#### 3a. `start_work`

**New MCP tool:**

```python
start_work(
    issue_id: str,
    assignee: str,
    target_status: str | None = None,   # explicit override; see §3c
    actor: str | None = None,
) -> IssueDict
```

Atomically: (a) optimistic CAS on `assignee` (same mechanism as
`claim_issue`), AND (b) advance status to the target. Single DB
transaction — either both land or neither does.

**Target status resolution:**

- If `target_status` is provided → use it directly (must be a valid
  transition from current status).
- If `target_status` is `None` → resolve via
  `WorkflowPack.canonical_working_status(type_name, current_status)`:

```python
class WorkflowPack:
    def canonical_working_status(
        self,
        type_name: str,
        current_status: str,
    ) -> str:
        """Return the unique `wip`-category status reachable from current.

        Raises AmbiguousTransitionError if multiple wip-category targets
        exist from current_status.
        Raises InvalidTransitionError if no wip-category target exists
        from current_status.
        """
```

On `AmbiguousTransitionError`: return `VALIDATION` with message naming the
candidate targets ("pick one via `target_status=`"). On
`InvalidTransitionError`: return `INVALID_TRANSITION`.

Explicit `target_status` override keeps `start_work` composable for packs
with complex workflows — no caller has to fall back to a separate tool.

#### 3b. Idempotency and concurrency — full 2×2

Behavior depends on (current category) × (assignee state):

| Current category | Caller owns claim | Claim free | Someone else owns |
|---|---|---|---|
| `open` | Transition to target. Return full issue. | Claim + transition atomically. Return full issue. | `CONFLICT` with `current_assignee` in details. |
| `wip` | Already-there no-op. Return full issue. | Claim (no transition needed). Return full issue. | `CONFLICT` with `current_assignee`. |
| `done` | `INVALID_TRANSITION` ("reopen first"). | `INVALID_TRANSITION`. | `INVALID_TRANSITION`. |

Safe for retry. Safe for concurrent callers — exactly one wins the claim.

#### 3c. `start_next_work`

```python
start_next_work(
    assignee: str,
    type: str | None = None,
    priority_min: int | None = None,
    priority_max: int | None = None,
    target_status: str | None = None,
    actor: str | None = None,
) -> IssueDict
```

Composes `claim_next + start_work`. Same atomicity guarantee, same target
resolution, same error semantics. Tie-break ordering for issue selection
is documented (see §3e).

#### 3d. CLI mirrors

- `filigree start-work <issue-id> --assignee <name> [--target-status <status>]`
- `filigree start-next-work --assignee <name> [--type X] [--priority-min N] [--priority-max N] [--target-status <status>]`

CLI name is `start-next-work` (not `start-next`) — matches MCP and removes
the "start the next *what*?" ambiguity.

#### 3e. `claim_next` / `start_next_work` tie-break

Documented selection order for "next" selection:

1. `priority` ascending (0 = critical first).
2. `created_at` ascending (oldest first within a priority tier).
3. `issue_id` ascending (deterministic tie-break for the rare collision).

This is also recorded in the MCP tool `description` and `docs/mcp.md`
so agents can reason about what they'll get.

#### 3f. `claim_issue` and `claim_next` stay

Intentionally. They are correct atomic primitives with niche uses (reserve
an issue without transitioning). Doc comments updated to point at
`start_work` / `start_next_work` as the usual compositions.

**No `complete_work`.** `close_issue(issue_id, reason=None)` is already a
single atomic op; adding a synonym creates duplication this release
removes elsewhere.

### 4. CLI parity

**Invariant after 2.0:** anything achievable via MCP is achievable via CLI.

**New CLI modules:**

- `src/filigree/cli_commands/observations.py`
  - Commands: `observe`, `list-observations`, `dismiss-observation`,
    `promote-observation`, `batch-dismiss-observations`.

- `src/filigree/cli_commands/files.py`
  - Commands: `list-files`, `get-file`, `list-findings`, `get-finding`,
    `update-finding`, `promote-finding`, `dismiss-finding`,
    `batch-update-findings`.

- `src/filigree/cli_commands/scanners.py`
  - Commands: `trigger-scan`, `trigger-scan-batch`, `get-scan-status`,
    `preview-scan`, `report-finding`, `list-scanners`.

Each is a thin click wrapper over the same core function the MCP handler
calls. No duplicated business logic. Registered via
`cli_commands/__init__.py`.

**Naming rule for new commands.** Primary name is kebab-case of the MCP
name, minus redundant `get-`/`list-` when a Unix idiom exists (`filigree
ready`, not `filigree get-ready`; but `filigree list-findings`, since
`findings` alone is noun-not-verb).

**MCP verb-noun aliases on existing CLI commands.** Bilingual CLI: existing
Unix-idiomatic names stay primary (`ready`, `blocked`, `stats`, `labels`,
`plan`, `transitions`, etc.), and MCP names register as click aliases
(`get-ready`, `get-blocked`, `get-stats`, `list-labels`, `get-plan`,
`get-valid-transitions`, etc.). **Aliases are permanent** — both forms
stay supported indefinitely. No future deprecation cycle. Committing to
permanence now prevents the 3.0 churn conversation.

**Shape alignment.** Every CLI command emits `--json` output in the exact
`BatchResponse` / `ListResponse` / `ErrorResponse` / `SlimIssue` shapes
from §1. Non-JSON output remains human-friendly. The JSON path is the
automation contract.

**Exit codes standardized:**

| Code | Meaning | Mapped `ErrorCode` |
|---|---|---|
| 0 | success (including empty results) | — |
| 1 | operational error | `PERMISSION`, `INVALID_TRANSITION`, `IO`, `NOT_FOUND`, `INVALID_API_URL`, `STOP_FAILED` |
| 2 | usage / validation error | `VALIDATION` |
| 3 | not initialized / schema mismatch | `NOT_INITIALIZED` |
| 4 | contention / conflict | `CONFLICT` |

Exit 4 for `CONFLICT` is separate so automated callers can profitably
retry. Anything at exit 1 is not retryable without operator intervention.

**Federation fallback path.** CLI `report-finding` accepts a Filigree-native
finding JSON (per the existing `POST /api/v1/scan-results` contract) on
stdin or via `--file`, calling the core ingest function directly — no HTTP
required on the caller side. This is the path any sibling emitter uses if
it lacks an HTTP client.

**Hooks.** `src/filigree/hooks.py` already shells out to `filigree …`;
`observe` as a CLI command closes the real capability gap the review
named.

### 5. Compact `get_issue` default

`include_files` defaults to `False`. Callers who want files pass
`include_files=True` explicitly. When the flag is false, the response omits
the `files` key entirely (not an empty list).

CLI mirror: `filigree show <id>` omits files by default; `--with-files`
opts in. `filigree files <id>` (existing) remains the "just give me the
files" command.

### 6. Schema-version-mismatch UX

**Problem observed in-session.** A `filigree` installed via `uv tool`
(v8) against a project DB upgraded to v9 produces a raw `ValueError`
stack trace from `FiligreeDB.initialize()` (`src/filigree/core.py:318`).
Operators get no guidance. The MCP dashboard dies silently.

**Change 1: new typed exception.** Replace `raise ValueError(msg)` at
`core.py:318` with `raise SchemaVersionMismatchError(installed=N,
database=M)`. Typed exception; structured fields; no string-match
fragility.

**Change 2: `filigree doctor` detects and guides.** Adds a check that
runs before the generic "can I open the DB?" check. On mismatch, output:

```
ERROR: installed filigree is schema v8 but this project's database is v9.
       Downgrade is not supported.
       Upgrade: uv tool install --upgrade filigree
       If this project was initialized by a newer filigree and you need
       to work on it with your current install, contact the project owner.
```

Exit code 3.

**Change 3: dashboard startup.** `dashboard.py:465` catches
`SchemaVersionMismatchError` specifically, logs the same guidance, exits 3
(no stack trace for operator-actionable errors).

**Change 4: MCP server — warm-but-degraded mode.** The MCP server *does*
launch on schema mismatch (so the agent's MCP connection succeeds) but
every subsequent tool call returns an `ErrorResponse` with `code:
NOT_INITIALIZED` and the same guidance text in the `error` message.
Rationale: a dead MCP server (connection refused) gives the agent no path
to diagnose the problem. A running server that answers every call with
"upgrade your tool" gives the agent an actionable message through the
protocol channel it's already using.

**Change 5: `filigree init` forward warning.** When `init` runs and the
*newly-created* DB uses a schema version strictly higher than any DB it
has previously opened (or when it migrates an existing DB to a new
version), it writes a `.filigree/INSTALL_VERSION` marker and emits:

```
WARNING: this filigree install is schema v9. If you run other filigree
         binaries against this project, make sure they're on v9 or newer.
         Older installs will refuse to open the database (see
         `filigree doctor`).
```

Cheap; prevents the senior-user "why did the dashboard just die?" confusion
after a local DB upgrade.

## Cross-product considerations

If sibling tools (Clarion, Wardline, other future consumers) adopt filigree
as their work-state surface, three contracts matter:

1. **`POST /api/v1/scan-results`** — existing Filigree-native finding intake
   endpoint. Response shape changes to the new `BatchResponse` envelope in
   this release. Any sibling currently consuming this endpoint updates in
   lockstep (one line per response shape). Clarion's `sarif import`
   translator is the known v0.1 consumer; others follow.

2. **`SlimIssue.issue_id`** — exposed via every endpoint returning embedded
   issue summaries (search, unblocked, ready, blocked). Sibling clients
   that pattern-match on this field need the rename applied in the same
   release window.

3. **`ErrorCode` values** — consumers branching on error codes see the
   consolidated values. The consolidation itself is a break even for
   unchanged codes, if a consumer tests for presence of one of the old
   retired strings.

**Coordination at release.** Any sibling consuming these three contracts
updates alongside filigree 2.0. No dual-accept window; version skew handled
bilaterally. None of the changes above depend on sibling adoption —
filigree 2.0 stands on its own as "MCP and CLI drifted, here's the clean
cut." The cross-product section exists so the coordination points are
explicit, not because they are the motivation.

## Migration and release

Filigree's primary consumer is Claude Code, which dogfoods through every
change and self-heals from API breaks by reading source. Polished migration
artifacts are mostly performative for this consumer profile.

**Required:**

- CHANGELOG `[2.0.0]` section listing all breaking changes (rename table,
  envelope shapes, error-code consolidation, default flip, dashboard error
  format, `start_work` addition, CLI parity additions, exit-code changes).
- Git commit messages that describe the breaking change in plain prose
  (`fix:` / `feat:` prefixes for the stage commits, `feat!:` for the final
  2.0 tag commit).
- Root and project CLAUDE.md / AGENTS.md updated to the new vocabulary
  (`issue_id`, `status`, `start_work`, `start-next-work`) in the Stage 6 PR.

**Optional but cheap:**

- Brief `docs/migration/2.0.md` listing the rename table only. No
  philosophy, no examples. Exists as a grep target for someone reading the
  CHANGELOG.

**Not doing:**

- Dual-accept compat shim.
- Deprecation-warning window in 1.x.
- Polished side-by-side example migration guide.

## Testing strategy

**Mechanical updates (caught by type checker):**

- Every fixture / test referencing `{"id": ...}` becomes `{"issue_id": ...}`.
- Every assertion on `response["closed"]`, `response["errors"]`,
  `response["results"]`, `response["count"]`, `response["updated"]` updates
  to `response["succeeded"]` / `response["failed"]`.
- Every handler using stringly-typed codes updates to `ErrorCode.*`.

**New tests — atomicity of composed ops:**

- Concurrent `start_work` from two threads on the same issue: exactly one
  succeeds, the other gets `CONFLICT`. Transition does not land if claim
  lost (verified by re-reading issue state after).
- Concurrent `start_next_work` with overlapping priority filters: each
  ready issue claimed by exactly one caller.
- `start_work` 2×2 idempotency table (§3b) → one test per cell.

**New tests — envelope invariants (parametrized):**

- For each issue-centric batch tool: response has exactly `{succeeded,
  failed}` + optional `newly_unblocked`, with `succeeded: list[SlimIssue]`
  in slim mode and `list[IssueDict]` in full mode.
- For each non-issue batch tool (`batch_update_findings`,
  `batch_dismiss_observations`): same shape with `T = FindingRecord` /
  `ObservationRecord` as appropriate.
- For every list tool: response has exactly `{items, has_more}` + optional
  `next_offset`.
- `failed[i].code` is always an `ErrorCode` member (not a raw string) —
  asserted per batch tool.

**New tests — CLI ↔ MCP parity (parametrized over new CLI commands):**

- Invoke CLI with `--json`; invoke equivalent MCP handler; assert response
  JSON is identical up to ordering.

**New tests — schema-mismatch path (three surfaces):**

- `doctor`: v9 DB fixture + mocked v8 install → exit code 3, guidance in
  stderr.
- dashboard startup: same fixture → clean exit 3, no stack trace in log.
- **MCP server warm-but-degraded**: same fixture → server launches; every
  tool call returns `ErrorResponse` with `code: NOT_INITIALIZED`.
- `filigree init` forward warning: fresh DB at new schema → warning
  emitted; `.filigree/INSTALL_VERSION` marker present.

**New tests — workflow-pack canonical working status resolution:**

- Pack with unique `open → wip` transition: `start_work` resolves it.
- Pack with multiple `wip`-category targets from `open`: `start_work`
  returns `VALIDATION` with candidates named (unless `target_status` given,
  in which case honored directly).
- Pack with no `open → wip`: `start_work` returns `INVALID_TRANSITION`.
- Explicit `target_status` overrides resolution in all three cases.

**Pre-push checklist** (from standing memory):

```bash
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
uv run mypy src/filigree/
uv run pytest --tb=short
```

Gate before the final 2.0 merge.

## Execution sequencing

Stage-by-stage PRs into the `2.0-project-management-extension` branch.
Each stage is independently reviewable and leaves the branch in a working
state.

### Stage 1 — Foundation contracts

Types-only PR. Extends `src/filigree/types/api.py`:

- `BatchResponse`, `BatchFailure`, `ListResponse` TypedDicts (Generic in
  `T`; type aliases for each concrete binding).
- `ErrorCode` StrEnum with the 9-member set.
- Updated `ErrorResponse` (flat shape, optional `details: dict`).
- New `SchemaVersionMismatchError` exception class.
- `AmbiguousTransitionError`, `InvalidTransitionError` exception classes
  for `canonical_working_status`.

No behavioral changes. Review checkpoint: types approved before anything
consumes them.

### Stage 2a — Error-code enum rollout

Replace every stringly-typed code across MCP tools, CLI commands, and
dashboard routes with `ErrorCode.*` members. No response-shape changes
yet — only the `code` field values and the flat `ErrorResponse` alignment.
Medium PR, low-risk mechanical substitution.

### Stage 2b — Batch and list envelope rollout

Rewrite every batch handler's return shape to `BatchResponse`. Rewrite
every list handler's return to `ListResponse`. Add `response_detail`
param to batch tools. Update all tests to assert the new shapes. Larger
PR but isolated from the code-name substitution in 2a.

**Stage 2 revert readiness.** The Stage 2b PR commit message documents
the exact response-shape change per tool so that a revert is a surgical
operation if a downstream consumer breaks catastrophically in dogfooding.
We do not expect to use this; we document it so that "what does rolling
back look like" is answered before we need it.

### 🔁 Dogfood checkpoint (between Stage 2b and Stage 3)

Before proceeding, run one manual agent session against the branch: a
real task (claim an issue, make an observation, batch-close a cluster,
list findings, etc.) through both MCP and CLI. Shape drift in response
bodies is only caught by tests *that assert the shape* — if any test file
stopped at response presence without key assertions, drift ships. Human
eyeball once.

Acceptance criteria: no unexpected `KeyError` in agent logs; all tools
return valid envelope shapes; at least one real cluster of work completed
through the branch without manual intervention.

### Stage 3 — Composed operations

Add `start_work`, `start_next_work`, and the `WorkflowPack.
canonical_working_status` helper. `target_status` override supported. MCP
tools + CLI commands + tests. Additive only.

### Stage 4 — CLI parity

Add `cli_commands/observations.py`, `files.py`, `scanners.py`. Register
MCP verb-noun aliases on existing CLI commands (permanent). Extend
`docs/cli.md`. Additive only.

### Stage 5 — Schema-mismatch UX

`SchemaVersionMismatchError` thrown from `core.py`. `doctor`, dashboard
startup, and MCP server (warm-but-degraded) catch and emit guidance with
standardized exit codes. `filigree init` forward-warning marker. Isolated
change set.

### Stage 6 — Renames, default flips, CLI arg-order alignment

Find-replace PR + one targeted arg-order fix:

- `id` → `issue_id`, `from_id` → `from_issue_id`, `to_id` → `to_issue_id`,
  `parent_id` → `parent_issue_id`.
- `SlimIssue.id` → `SlimIssue.issue_id`.
- `state` → `status` in MCP tool names and CLI command names.
- `get_issue.include_files` default flips to `False`; add `--with-files`
  to `filigree show`.
- `add-label` and `batch-add-label` align on `<label> <ids...>` positional
  order.
- CLAUDE.md / AGENTS.md update in the same PR.

Type checker catches most drift; grep-sweep fills in the rest.

### Stage 7 — Release docs

CHANGELOG `[2.0.0]` section. Optional short `docs/migration/2.0.md`. Tag
and merge. Ship.

### Rationale for this order

Contracts first (Stage 1), then code-name substitution (Stage 2a), then
shape substitution (Stage 2b), then a dogfood gate, then additive features
(Stages 3–5), then cutover (Stage 6), then release hygiene (Stage 7). Each
stage leaves the branch working. The biggest breakage (Stage 6 renames)
lands late so earlier stages aren't constantly re-merging against a
shifting name space. The 2a/2b split keeps each PR reviewable; the dogfood
gate catches shape drift tests alone can miss.

## Open questions

**Deferred / future considerations (not in 2.0 scope):**

- **`retryable: bool` on `ErrorResponse`.** The `IO` consolidation loses
  the transient-vs-permanent distinction. If retry semantics matter in
  practice, add a boolean flag on the error response in a later release —
  orthogonal to the code enum. Not adding now because no caller currently
  branches on it.
- **Batch `get_comments`.** If triaging 10 issues means 10 round trips to
  pull comments, a batch variant would help. Out of scope for 2.0; worth a
  separate issue if it bites.
- **"Get my work" shortcut** (`filigree mine` / `mcp__filigree__my_work`).
  Sugar over `list_issues --assignee=<me>`. Nice but not in the ranked 7.
- **`create_plan`'s 3-level nested schema.** Structural complexity not
  addressable by rename or envelope; separate spec if promoted.

**Resolved during brainstorming and senior-user review:**

- Scope: all 7 recommendations + follow-throughs from review pass.
- Migration strategy: hard cutover, no dual-accept.
- CLI parity scope: full parity.
- Sequencing: staged by concern, infrastructure-first, with dogfood gate.
- `IO` stays unified (retryable flag is the future addition if needed).
- `BatchResponse` gets `response_detail: "slim" | "full"` opt-in.
- `start_work` gets explicit `target_status` override.
- MCP server uses warm-but-degraded mode on schema mismatch.
- `CONFLICT` gets its own exit code (4).
- `claim_next` tie-break documented.
- `filigree init` emits a forward-warning on new schema versions.
- Bilingual CLI aliases are permanent (no future deprecation cycle).
- `batch-add-label` positional-arg alignment happens in Stage 6, not
  deferred.
