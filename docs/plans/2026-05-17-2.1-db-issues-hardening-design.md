# Filigree 2.1 — Issues-Layer Hardening Design

**Date:** 2026-05-17
**Author:** Claude (Opus 4.7) panel synthesis, with John Morrissey
**Status:** Implemented in 2.1.0 (see CHANGELOG)
**Source panel:** db_issues.py review (solution-design, systems-thinking, python-engineering, coverage-gap-analysis, threat-analysis, embedded-database, silent-failure-hunt)
**Related:** PR #41 (2.0.4 — core.py worktree hardening, shipping next)

## Summary

Filigree 2.1 closes a tight cluster of correctness, security, and
concurrency findings in the issues data layer (`db_issues.py`, the busiest
write surface in the codebase) plus a small set of items deferred from the
2.0.4 core.py hardening. The findings fall into five themes:

1. **Optimistic-lock holes.** `update_issue` reads-then-writes without a CAS
   guard on assignee; every claim-aware write (close, batch_close,
   batch_update) inherits the race. Audit-trail attribution can flip
   silently between concurrent agents.
2. **Audit-trail silent loss.** `_record_event` `INSERT OR IGNORE` swallows
   same-second collisions. `reopen_issue` writes the `reopened` event in a
   separate transaction wrapped in `try/except` that suppresses failure.
   Both corrupt the audit signal `undo_last`, timelines, and downstream
   consumers depend on.
3. **HTTP attack surface.** `force=true` is plumbed through `POST
   /api/batch/close` on a no-auth localhost dashboard, with caller-chosen
   `actor`. `WrongProjectError` returns absolute prefixes over the wire.
   `get_issue` skips prefix enforcement and is reachable across projects in
   server-mode multi-tenancy.
4. **SQLite transaction discipline.** Every write begins implicit-DEFERRED
   and upgrades to writer on first UPDATE; under multi-agent contention the
   upgrade can fail past `busy_timeout=5000` and surface a raw
   `OperationalError`. `_begin_immediate` already exists in `db_base.py`
   but is never called from `db_issues.py`.
5. **Workflow surface drift (systemic).** `_skip_transition_check` and
   `force` exist because backward edges (reopen, release-revert,
   force-close) are not first-class template concepts. Each new
   backward-edge operation adds another bypass call site rather than
   declaring the edge.

Plus three deferred-from-2.0.4 items: the submodule-inside-worktree test
gap, the `ForeignDatabaseError` remediation hint misdirection, and
`dashboard.py` structured-log redaction.

This release is **scoped breaking** at the audit-trail and workflow-surface
boundaries. Wire shapes that are user-visible (CLI output, MCP envelope
fields) stay stable; internal kwargs (`_skip_transition_check`), HTTP body
shapes for `force`, and `_record_event` collision behaviour change.

## Context

### Why this release, why now

PR #41 fixed worktree-discovery hardening (2.0.4); CI is green and the
release-day-or-two cadence holds. The panel that audited `db_issues.py`
afterward surfaced a more concerning cluster than `core.py` did — three
items in the **critical** band cross security, audit-integrity, and
optimistic-concurrency lines respectively. None are exploitable in casual
single-agent CLI use, all are exploitable (or already misbehaving) under
the multi-agent concurrency the system was built for. Two reviewers
independently flagged "shifting the burden" / "fixes that fail" archetypes
on the workflow surface, which a tactical 2.0.5 cannot reach.

A 2.1 is the right cut because:

- The critical-band items want coordinated changes across `db_issues.py`,
  `db_events.py`, the dashboard routes, and the schema (event dedup index
  needs a sequence column or a millisecond timestamp). A bug-fix release
  cannot make schema changes safely.
- The "shifting the burden" pattern (backward edges in templates) is a
  template-format change with migration implications — a minor-version
  cut is the right venue.
- The release notes can be written from one source rather than four
  point-release patches that drift.

### Non-goals

- **No new features.** No new tools, verbs, or fields beyond what these
  fixes require.
- **No `BatchResponse` / `ListResponse` envelope changes.** The 2.0
  unified-surface contract holds.
- **No CLI vocabulary changes.** `id` / `status` / batch-result shapes
  stay as 2.0 shipped them.
- **No general refactor of `IssuesMixin`.** Solution-architect's H3
  god-class observation (split into `ClaimsMixin` / `SearchMixin`) is
  filed as a follow-up; refactoring 2108 lines mid-fix invites regression.
- **No new project-isolation model.** `_check_id_prefix` remains the
  single guard; 2.1 only closes the gaps where it's missing or
  bypass-able.
- **No federation / Loom protocol additions.** Out of scope.

## Design

Findings are numbered by phase. Within each phase, ordering reflects
dependency: earlier items unblock later ones.

### Phase 0 — Critical correctness (lands first; can ship as 2.0.5 if 2.1 slips)

These four are exploitable today under multi-agent load. They are small
enough to land independently; if 2.1 slows for any reason they should
ship as 2.0.5.

#### §0.1 `update_issue` — atomic CAS on assignee

**Panel:** silent-failure C1, solution-architect C1, threat T-001, embedded-db H1.

**Current** (`db_issues.py:638–880`):
```python
current = self.get_issue(issue_id)
_check_expected_assignee(issue_id, expected_assignee, current.assignee, ...)
# ... validation ...
self.conn.execute(f"UPDATE issues SET {', '.join(updates)} WHERE id = ?", params)
```
Two-step read-then-write. Another agent's UPDATE between the two silently
wins audit-trail attribution, transition validation, and field state.

**Target:** add `AND assignee = ?` to the UPDATE when an assignee was
observed; on `rowcount == 0`, re-read and raise `ClaimConflictError` (see
§0.3). Pattern matches `claim_issue:1080`, `heartbeat_work:1332`,
`reclaim_issue:1438` — all three already do this; only `update_issue` and
its derivatives (`close_issue`, `batch_close`, `batch_update`) are out of
step.

**Risk:** medium. Every write surface flows through here; misimplementation
breaks every test. Mitigated by mechanical mirror of an existing pattern.

**Test plan:**
- `test_update_issue_cas_rejects_concurrent_reassign` — two `ThreadPoolExecutor` workers, one updates, other reassigns; assert exactly one wins, loser sees `CONFLICT`.
- `test_close_issue_inherits_cas` — same shape via `close_issue`.
- `test_batch_close_per_item_cas` — middle item loses the race; assert items before/after still commit.

#### §0.2 `_record_event` — make collisions visible

**Panel:** silent-failure C3.

**Current** (`db_events.py:96–100`, schema `db_schema.py:64–66`): the
events table carries a UNIQUE dedup index across `(issue_id, event_type,
actor, old_value, new_value, created_at)`. `_record_event` writes with
`INSERT OR IGNORE`. Same-actor heartbeats in the same ISO-second, batch
ops sharing a single `_now_iso()`, and tight transition_warning loops all
collide and silently lose one event.

**Target:** two-part fix.

1. **Schema (migration):** extend the dedup tuple with a monotonic
   per-event sequence column (`event_seq INTEGER NOT NULL`) or migrate
   `created_at` to millisecond precision. The sequence column is
   preferred — millisecond timestamps still collide under bursty writes
   and depend on system clock behaviour.
2. **Code:** `_record_event` switches from `INSERT OR IGNORE` to plain
   `INSERT`; if the dedup constraint ever fires (true duplicate, not
   accidental same-second), raise an `IntegrityError` that the caller's
   transaction can roll back. Capture `cursor.rowcount` defensively as
   a belt-and-braces check.

**Migration:** schema version bump. The migration runs once on first
2.1 connect; adds the `event_seq` column with default 0 for historical
rows; the new UNIQUE index uses `(issue_id, event_type, actor,
old_value, new_value, created_at, event_seq)`. Forward-only per
2.0 non-goal.

**Risk:** medium-high — touches every event-emitting code path. Coverage
matters: every mixin that calls `_record_event` needs the same
treatment.

**Test plan:**
- `test_heartbeat_bursts_record_every_event` — fire 10 heartbeats in a tight loop, assert all 10 land in the events table.
- `test_record_event_sequence_monotonic` — same actor, same type, same second, multiple emissions; assert sequence increments.
- `test_record_event_dedup_still_fires_on_true_duplicate` — same row INSERTed twice → IntegrityError raised, caller transaction rolls back.

#### §0.3 Typed `ClaimConflictError` — replace string-matched dispatch

**Panel:** python-engineer C1, silent-failure H6.

**Current:** Five sites in `db_issues.py` (lines 1290, 1688, 1793, 1830,
1865) classify `ErrorCode.CONFLICT` by inspecting exception message text:
```python
code = ErrorCode.CONFLICT if "expected" in msg and "assigned to" in msg else ErrorCode.VALIDATION
```
This violates the project's own CLAUDE.md ("switch on `code`, not
message text"). Rewording the conflict message anywhere downstream
silently downgrades CONFLICT to VALIDATION.

**Target:**
- Add `class ClaimConflictError(ValueError)` in `db_base.py` or
  `types/api.py`.
- `_check_expected_assignee` raises `ClaimConflictError`, not bare
  `ValueError`.
- `claim_issue` / `release_claim` / `reclaim_issue` / `heartbeat_work`'s
  CAS-failure paths raise `ClaimConflictError` (they currently raise
  `ValueError` via the same message-text dispatch).
- The five batch dispatch sites catch `ClaimConflictError` first and
  emit `ErrorCode.CONFLICT`; broader `ValueError` continues to map to
  `VALIDATION`.

**Backward-compatibility:** `ClaimConflictError` extends `ValueError`, so
existing callers catching `ValueError` still work. The `WrongProjectError`
sibling (also `ValueError`) gets the same treatment in §0.4 — both subclass
checks must precede the generic handler.

**Risk:** low. Mechanical; tests should pin the new exception type at
every site that raises it.

**Test plan:**
- `test_claim_conflict_raises_typed_exception` — every CAS-failure path.
- `test_batch_close_reports_conflict_for_claim_conflict` — verify
  `ErrorCode.CONFLICT` in the batch response when one item loses a
  claim race.

#### §0.4 `WrongProjectError` precedence in batch handlers

**Panel:** silent-failure H7.

**Current:** `WrongProjectError` extends `ValueError`. Every batch handler
catches `ValueError` as a per-item failure. A batch pointed at the wrong
DB produces N "validation" failures with no signal that the caller is in
the wrong project — exactly the silent foreign-DB-mutation that
`core.py`'s anchor discovery was hardened against.

**Target:** every batch handler (`batch_close`, `batch_update`,
`batch_add_label`, `batch_remove_label`, `batch_add_comment`,
`release_my_claims`, plus the `_batch_with_transition_errors` helper)
catches `WrongProjectError` (and `ClaimConflictError` per §0.3) before
the generic `ValueError`. `WrongProjectError` aborts the whole batch
with a single envelope-level error, not per-item failures.

**Risk:** low. Mechanical exception-ordering fix; tests pin both
single-issue and batch surface.

**Test plan:**
- `test_batch_close_foreign_prefix_aborts_batch_not_per_item` — one foreign-prefix ID in a batch raises immediately, no per-item commits.
- `test_release_my_claims_foreign_prefix_aborts` — same for release_my_claims.

### Phase 1 — Security & isolation

These three sit at the HTTP boundary. They are not exploitable from a
single-user CLI session, but the dashboard exposes them on localhost with
no authentication, and any local process — including a browser tab the
user opens, or an agent running with the user's privileges — can drive
them.

#### §1.1 Remove `force` from HTTP batch routes

**Panel:** threat T-002.

**Current:** `POST /api/batch/close {force: true}` and `POST
/api/loom/batch/close {force: true}` are reachable on `127.0.0.1:8377`
with no authentication. `force=true` bypasses the template's transition
table and every hard-enforcement field gate (e.g., a `tests_passed`
field whose value must be `true` before close). Any local process can
rage-close any issue.

**Target:** remove `force` from both HTTP routes. The CLI `--force` and
MCP `force=` arguments stay (they require a local CLI invocation or an
MCP connection, which presupposes the user's intent). The HTTP `force`
parameter is replaced by `503` if a caller attempts to use it on a
configuration that lacks the new opt-in flag.

**Opt-in flag (for future use):** a dashboard startup option
`--allow-http-force-close` (default off) re-enables the HTTP parameter
behind an explicit operator choice. Use case: a release script that
shells into the dashboard's project to clean up stuck batches. Default
off so the threat surface is closed for normal use.

**Audit:** every `force=true` transition emits a distinct event type
(`transition_forced`) in addition to the normal `status_changed` event,
so the audit trail flags bypasses. Implemented in `update_issue` when
`_skip_transition_check=True`.

**Risk:** low. Wire-shape change. Affects any HTTP caller using `force`;
none in the codebase. Document in CHANGELOG as a breaking HTTP API
change.

**Test plan:**
- `test_http_batch_close_force_returns_400` — POST with `force=true` returns 400 with `code=VALIDATION` and a message pointing at the CLI/MCP alternatives.
- `test_http_batch_close_force_with_opt_in_works` — same call with `--allow-http-force-close` set succeeds.
- `test_forced_transition_emits_audit_event` — CLI `--force` close emits both `status_changed` and `transition_forced`.

#### §1.2 `WrongProjectError` redaction on wire surfaces

**Panel:** threat I-3, deferred from 2.0.4.

**Current:** `WrongProjectError.__str__` embeds the foreign project's
prefix. Over the multi-project dashboard (`/api/p/{key}/...`), guessing
or probing IDs lets a caller enumerate prefixes of other registered
projects. The dashboard already redacts paths for `ForeignDatabaseError`
(see PR #41) — `WrongProjectError` was missed.

**Target:** mirror the `ForeignDatabaseError.safe_message` pattern from
PR #41. Add `WrongProjectError.safe_message` returning a redacted
variant ("Issue ID does not belong to this project"). HTTP and MCP
serialise the redacted variant; CLI/stderr/doctor keep the rich
message for diagnostic value. Symmetric with §6 of PR #41.

**Risk:** low. Pattern is established. One additional class, two call
sites updated.

**Test plan:**
- `test_wrong_project_error_safe_message_omits_prefix`.
- `test_http_404_for_foreign_id_does_not_leak_prefix` — full integration via dashboard.

#### §1.3 `get_issue` prefix enforcement in server-mode HTTP

**Panel:** threat T-003, E-1.

**Current:** `get_issue` (`db_issues.py:520–524`) deliberately skips
`_check_id_prefix`. The comment documents this is intentional ("reads
tolerate foreign IDs"). In server mode (`/api/p/{key}/issue/{id}`), a
caller hitting the wrong project key can read another project's issue
body, comments, labels, and events.

**Target:** enforce prefix at the HTTP route boundary in server mode,
not in `get_issue` itself. The data-layer comment about read tolerance
stays valid for CLI/MCP local use (an agent operating in one project
should still be able to read an ID it inherited from another context,
e.g. a copy-pasted reference). The dashboard route layer is the right
place to enforce the multi-tenant boundary — it already knows the
project key and can require the requested ID's prefix to match.

**Implementation:** add a small helper in
`dashboard_routes/issues.py` (and adjacent route files) that checks
the requested ID's prefix against `db.prefix` at every read endpoint.
Mirror the per-route `_check_id_prefix` calls already present on
writes. The helper lives once and is imported by all reads.

**Risk:** low–medium. Touches every issue read route. Mitigated by
extracting into one helper and a route-level decorator pattern.

**Test plan:**
- `test_dashboard_server_mode_blocks_cross_project_read` — register two projects A and B in the dashboard; GET `/api/p/A/issue/<B-id>`; assert 404 with redacted message.
- `test_get_issue_cli_still_tolerates_foreign_id_locally` — pin the design intent at the data-layer (so the route-level enforcement isn't accidentally pushed into `get_issue`).

#### §1.4 `actor` is sanitized but unverified — document and bound

**Panel:** threat T-001, T-010.

**Status:** the full fix (verify actor against transport-layer identity)
is **out of scope for 2.1.** The threat model is "agents tell the truth
when not prompt-injected" — actor verification would require a dashboard
session-token mechanism that doesn't exist and is a multi-release piece
of work.

**Target for 2.1 (minimal):**
- Document the threat model explicitly in
  `docs/architecture/decisions/` as an ADR: actor strings are
  identifiers, not authentications; the audit trail records claims, not
  proofs.
- Cap `actor` length (already done via `sanitize_actor` per the threat
  report) and verify it's enforced at every entry point — CLI, MCP, HTTP.
- Filed as a 2.2+ work package: actor verification via session token
  on dashboard, MCP transport identity for stdio servers.

**Risk:** documentation-only for 2.1. The follow-up is a real piece of
work that wants its own design pass.

### Phase 2 — SQLite transaction discipline

#### §2.1 `_begin_immediate` on every write path

**Panel:** embedded-db H1, H2.

**Current:** Python's `sqlite3` opens transactions implicitly at the first
write under `isolation_level="DEFERRED"`. Every write in `db_issues.py`
relies on this. `_begin_immediate` exists in `db_base.py` but is never
called from `db_issues.py`. Under multi-agent contention the
deferred-then-write upgrade can fail past `busy_timeout=5000` and
surface a raw `OperationalError` to the caller.

**Target:** every public write method in `db_issues.py` calls
`_begin_immediate(self.conn, "method_name")` before the first write
statement. The method's existing `try/except: rollback; raise` block
becomes the natural transaction boundary.

A small decorator simplifies adoption:
```python
@_in_immediate_tx("create_issue")
def create_issue(self, ...): ...
```
The decorator wraps with `_begin_immediate` + commit-on-success +
rollback-on-exception, replacing the per-method boilerplate. Methods
that compose with other writes (e.g. `start_work` calls `claim_issue`
then `update_issue`) pass `_skip_begin=True` to nested calls so the
outer decorator's transaction is preserved.

**Busy-retry layer:** small `@_retry_busy(attempts=3, base=0.05)`
decorator catches `sqlite3.OperationalError` where `"database is
locked"` or `"database is busy"` appears in `str(exc)`, retries with
exponential backoff, and re-raises after exhaustion. Applied via the
same decorator stack:
```python
@_retry_busy
@_in_immediate_tx("create_issue")
def create_issue(self, ...): ...
```

**Risk:** medium-high. Touches every write method (~15 methods in
`db_issues.py` alone, plus parallel work in `db_files.py`,
`db_planning.py`, etc.). Subtle bugs in the decorator stack are
catch-all expensive. Recommended approach: land the decorator stack with
tests, then migrate methods incrementally one PR per mixin, with the
mixin-level test suite as the smoke gate.

**Sequencing:** lands AFTER §0.1 (CAS on update_issue). The CAS check
must be inside the IMMEDIATE transaction; doing them in the other order
exposes a brief window where the deferred-then-write race is wider than
it is today.

**Test plan:**
- `test_immediate_transaction_decorator_rolls_back_on_exception`.
- `test_busy_retry_decorator_transparently_recovers` — simulate
  `OperationalError("database is locked")`, assert retry succeeds.
- `test_busy_retry_decorator_re_raises_after_exhaustion`.
- Re-run the full `db_issues` test suite under simulated contention
  (multiple `ThreadPoolExecutor` workers) and assert no SQLITE_BUSY
  raises to test code.

#### §2.2 `start_work` / `start_next_work` — shorten the held-writer window

**Panel:** embedded-db M2.

**Current** (`db_issues.py:1535–1656`): `claim_issue(_commit=False)` opens
the transaction, then `update_issue` does template lookups
(`tpl.reachable_working_status()` is pure-Python but non-trivial) inside
the open writer transaction. With many concurrent `start_next_work`
calls this lengthens the held-writer window for every waiting agent.

**Target:** resolve `target_status` BEFORE calling `claim_issue`. Held
writer window becomes (claim UPDATE) + (status UPDATE) + (event INSERTs)
+ COMMIT only. Refactor is local; signature unchanged.

**Risk:** low. Single-method refactor with a clear before/after.

**Test plan:** existing `start_work` tests + benchmark that pins the
held-writer time on a synthetic contention case.

#### §2.3 Lease-expiry check moved to SQL

**Panel:** embedded-db H4 (low).

**Current:** `get_stale_claims` (`db_issues.py:1356–1396`) reads all
assigned non-done rows then filters in Python via
`_parse_issue_timestamp`. At N issues × M polling agents this is O(N×M).

**Target:** push the expiry check into the SQL WHERE clause:
```sql
WHERE assignee != ''
  AND claim_expires_at IS NOT NULL
  AND datetime(claim_expires_at) <= datetime('now')
```
The application-side parse stays as a fallback for the (now-rare)
`claim_expires_at IS NULL` legacy rows.

**Risk:** low.

**Test plan:** `test_get_stale_claims_sql_filter_matches_python_filter` —
parameterised over a mix of expired / fresh / NULL rows.

### Phase 3 — Audit-trail integrity

#### §3.1 `reopen_issue` — single transaction

**Panel:** solution-architect H1, python-engineer C2, silent-failure C2.

**Current** (`db_issues.py:980–1009`): three separate commits — status
change via `update_issue`, then unguarded fields-cleanup UPDATE+commit,
then try/except-suppressed `reopened` event commit. A crash between
commits #1 and #3 leaves the issue reopened with no `reopened` event —
the signal `undo_last` depends on.

**Target:** fold into one transaction. Record the `reopened` event,
execute both UPDATEs, then `self.conn.commit()` once. The existing
`except Exception: rollback; raise` guard wraps the whole block. The
"reopened" event no longer needs special-case swallowing.

**Sequencing:** depends on §2.1 (`_begin_immediate` on writes) being
landed for the related methods; otherwise the single-transaction
guarantee is "DEFERRED-on-first-write," which has the lock-upgrade
problem.

**Risk:** low–medium. Touches a single method, but the audit-event
ordering matters and has dependents (`undo_last`).

**Test plan:**
- `test_reopen_issue_atomic_under_event_failure` — inject a failure into `_record_event` after the status UPDATE; assert the issue is still closed and no `reopened` event exists.
- `test_reopen_issue_single_transaction` — pin that `self.conn.commit()` is called exactly once.

#### §3.2 `release_claim` — atomic revert or compensate

**Panel:** solution-architect C2, silent-failure H9.

**Current** (`db_issues.py:1167–1208`): release commits at line 1192,
then a separate `update_issue` call reverts `wip → open`. If revert
raises (template missing, transition validator change, race), the
assignee is durably cleared but the caller sees the revert exception —
they cannot tell "release succeeded, revert failed" from "release
failed." Issue is left orphaned: unassigned, still in wip, invisible to
`get_ready`.

**Target:** two options, both acceptable; pick at implementation time:

1. **Atomic option:** compute the revert target before the release;
   issue a single UPDATE setting both `assignee = ''` and
   `status = <target>` in one statement; one commit.
2. **Compensating option:** catch the revert exception, record an
   `orphaned_in_wip` warning event, append a `data_warning` to the
   returned `Issue`, and return successfully. The orphan is then visible
   in the audit trail and via a warning-anchored CLI/MCP query.

Recommend the atomic option; the compensating option is fallback if
the SQL is too gnarly for one UPDATE (it isn't).

**Risk:** low. Single method.

**Test plan:**
- `test_release_claim_revert_is_atomic` — inject a revert failure; assert release does not commit either.
- `test_release_claim_emits_one_commit` — pin the transaction count.

#### §3.3 `_safe_fields_json` — refuse to launder corrupt JSON

**Panel:** silent-failure C4.

**Current** (`db_issues.py:857`): `current.fields` is a `_ParsedJson`
dict subclass carrying `_filigree_corrupt = True` when the row's JSON
was unparseable. `merged = {**current.fields, **fields}` produces a
plain dict — flag stripped. The subsequent `json.dumps(merged)`
durably overwrites the corrupt-but-recoverable bytes with `{}` plus
user fields. The `fields_changed` event's `old_value` records `{}` —
corruption laundered out of the audit trail.

**Target:** check the flag on `current.fields` before merging. If set,
refuse to update unless an explicit `force_overwrite_corrupt=True`
kwarg is passed. The `force_overwrite_corrupt` path emits a distinct
`corrupt_fields_overwritten` event capturing the original raw bytes
in `old_value`, preserving the audit signal.

**Risk:** low. Single helper with a clear guard.

**Test plan:**
- `test_update_issue_refuses_corrupt_fields_merge`.
- `test_update_issue_force_overwrite_corrupt_emits_event_with_raw_bytes`.

#### §3.4 `create_issue` — prefix-check `parent_id` and `deps`

**Panel:** silent-failure H8.

**Current** (`db_issues.py:401–518`): every other mutation calls
`self._check_id_prefix(issue_id)`. `create_issue` generates its own
ID, so the new row is fine — but `parent_id` and `deps` are
caller-supplied IDs that get FK-linked into the `issues` /
`dependencies` tables with no prefix validation. If a `parent_id`
happens to collide with a local row, the linkage is established
silently in the wrong project.

**Target:** call `_check_id_prefix(parent_id)` and
`_check_id_prefix(dep)` for each `dep` at the top of `create_issue`,
before any database writes.

**Risk:** trivial.

**Test plan:**
- `test_create_issue_rejects_foreign_parent_id`.
- `test_create_issue_rejects_foreign_dep_id`.

### Phase 4 — Workflow surface (systemic)

#### §4.1 Backward edges as first-class template concepts

**Panel:** systems-thinking primary archetype ("Shifting the Burden"),
solution-architect H3 partial.

**Current:** `_skip_transition_check=True` is invoked from four sites
(`reopen_issue`, `close_issue(force=True)`, `release_claim`,
`_rollback_uncommitted_start_claim`). Each represents a backward edge
that the workflow template can't express, so the code bypasses
validation. Every new backward-edge operation adds another bypass call
site rather than declaring the edge.

**Target:** add `reverse_transitions` (or `backward_edges` — pick name
at implementation time) to the workflow template YAML. Each backward
edge declares the same shape a forward transition does: source state,
target state, optional field gates, optional event type.

Implementation:
- Template loader recognises the new key and validates it the same way
  forward transitions are validated.
- `update_issue` accepts a `backward: bool` parameter (default false)
  that routes the transition through the backward-edge validator
  instead of the forward one.
- `_skip_transition_check` is removed.
- `reopen_issue`, `release_claim`'s revert, and `close_issue(force=True)`
  consume the new mechanism.

**Migration:** workflow packs ship with explicit backward-edge
declarations for the canonical reopen / force-close / release-revert
cases. Custom packs may need updating; the template loader provides a
clear error message naming the missing edge. A `filigree validate` pass
in `2.1.0-beta` (or whatever pre-release name fits the team's
conventions) catches stale packs early.

**Risk:** high. Template-format change with migration implications. The
panel's systems-thinker noted this is the highest-leverage intervention
(Meadows level 10 — structural). Worth doing once, properly.

**Sequencing:** lands LAST in 2.1 because every preceding fix can use
`_skip_transition_check` in the interim. Removing the bypass is the
final step.

**Test plan:**
- `test_template_backward_edge_declaration_validates`.
- `test_reopen_uses_backward_edge_not_skip_check` — `_skip_transition_check` removed; reopen still works because the template declares `closed → open` as a backward edge.
- `test_missing_backward_edge_raises_named_error` — custom pack without a needed backward edge fails with a clear message.

#### §4.2 `close_issue` — surface `valid_transitions` on single-issue failure

**Panel:** solution-architect H2.

**Current:** the 2.0.2 decision to remove auto-resolve in `close_issue`
left an asymmetry — `batch_close` enriches `INVALID_TRANSITION` with
the failing item's `valid_transitions`; single-call `close_issue`
raises a `ValueError` with a string saying "use `get_valid_transitions()`".
Every single-issue caller needs a second round-trip.

**Target:** raise a typed `InvalidTransitionError(ValueError)` carrying
the failing issue's `valid_transitions` as a typed attribute. The batch
helper unwraps the attribute into its existing envelope; the single-call
path lets callers introspect without a second round-trip. HTTP / MCP
serialise the attribute into the error envelope.

**Risk:** low–medium. Affects error-envelope shape. Backward-compatible
because the new class still subclasses `ValueError`.

**Test plan:**
- `test_close_issue_invalid_transition_carries_valid_transitions`.
- `test_batch_close_unwraps_invalid_transition_attribute`.

### Phase 6 — Diagnostic & UX hardening (deferred from 2.0.4)

#### §6.1 `ForeignDatabaseError` remediation hint

**Source:** PR #41 out-of-scope item; solution-architect M7 from core.py panel.

**Current:** `ForeignDatabaseError.__init__` produces `cd {git_boundary}
&& filigree init` as remediation. When `git_boundary` is a malformed
`.git` file (not a real repo root), the user follows the hint, creates
a conf, and the next invocation still fails — the conf doesn't fix
anything because the malformed `.git` is the real problem.

**Target:** classify `git_boundary / ".git"` (directory / file-with-valid-worktree-pointer / something-else) and adjust the remediation. At minimum append: "If `{git_boundary}/.git` is malformed, fix or remove it before running `filigree init`."

**Test plan:**

#### §6.2 `dashboard.py` structured-log redaction

**Source:** PR #41 out-of-scope item; threat analyst I-1.

## Cross-cutting concerns

### Concurrency-test infrastructure

Several Phase 0 / Phase 1 / Phase 2 items want true concurrent-write
tests against a single SQLite file. The existing test corpus uses
`_InterceptingConnProxy` to simulate races via mid-transaction
intercepts; that pattern works for TOCTOU-style tests but not for
genuine `OperationalError("database is locked")` reproduction.

A new fixture `concurrent_db_workers(n: int) -> list[FiligreeDB]` opens
N connections to the same on-disk DB and returns them to the test for
`ThreadPoolExecutor` orchestration. Lives in `tests/conftest.py`.
Lands as part of §0.1 (the first item that needs it).

### Migration strategy

The dedup-index change in §0.2 is the only schema change. Forward-only
per the 2.0 non-goal. Migration test (`tests/install/test_migrate.py`
or local equivalent) pins:
- Existing rows survive the migration.
- `event_seq` defaults sensibly for historical events (0 is fine —
  collision was already silent).
- Post-migration, new events get unique `event_seq` values.

### Audit-trail compatibility

Phase 3 changes the shape of two events:
- §0.2 — `_record_event` raises on dedup-constraint violation instead
  of silently ignoring. Callers that previously suppressed these never
  saw the suppressed events anyway, so the user-visible surface is
  unchanged; the migration only catches the events that were always
  meant to be recorded.
- §3.3 — new `corrupt_fields_overwritten` event type. Consumers (the
  dashboard timeline, `undo_last`) get a new kind they previously
  couldn't render. Add rendering as part of the phase.

§1.1's `transition_forced` event is also new; same treatment.

## Migration and release

### Versioning

This is a minor version (2.1.0). The user-visible breaking changes are:
- HTTP `POST /api/batch/close {force: true}` returns 400 by default
  (§1.1). Opt-in restores it.
- The events table gains an `event_seq` column (§0.2). Schema version
  bumps.

Both are acceptable for a minor version per the 2.0 contract: HTTP
batch routes are not part of the documented "CLI / MCP stable surface,"
and schema migrations are forward-only by policy.

### Release sequencing

Recommend:
- **2.0.4** — PR #41 lands as-is (worktree hardening). Already done.
- **2.0.5 (optional)** — Phase 0 items (`update_issue` CAS, typed
  `ClaimConflictError`, `WrongProjectError` precedence, `_record_event`
  collision visibility *without* the schema change — just `rowcount`
  detection + log + `data_warnings`). If shipped, 2.1.0 then folds the
  schema change for proper raise-on-collision.
- **2.1.0** — Phases 1–4 + Phase 5 tests + Phase 6 polish. Phase 4
  (backward edges) is the largest, lands last in 2.1.0, and is the
  point at which `_skip_transition_check` is removed.

If 2.0.5 isn't worth the release cadence, Phase 0 ships as part of
2.1.0 alpha.

### Documentation

- `CHANGELOG.md` — section per phase.
- `docs/architecture/decisions/` — one ADR for the actor-identity threat
  model (§1.4) and one for the backward-edge template feature (§4.1).
- `docs/architecture/templates.md` — backward-edge reference (§4.1).
- `src/filigree/data/instructions.md` — update to reflect typed
  exceptions (§0.3, §0.4, §4.2) so agents stop catching bare
  `ValueError`.

## Open questions

1. **Do we ship Phase 0 as 2.0.5?** The cadence question is operational,
   not technical. If we want the security and audit-trail items in
   users' hands fast, 2.0.5 is the right cut. If we want one coherent
   release with one set of release notes, fold into 2.1.0.

2. **`event_seq` vs millisecond timestamp** (§0.2). The spec prefers
   `event_seq` because it's clock-independent, but a millisecond
   timestamp is a smaller migration. Decision point at implementation
   time; both are correct.

3. **`force_overwrite_corrupt` kwarg name** (§3.3). Bikeshed — any
   reasonable name works as long as the audit event records what
   happened.

4. **Backward-edge template syntax** (§4.1). Match forward-transition
   YAML exactly (single key `reverse_transitions:`) or introduce a
   richer DSL? Recommend the former — symmetry beats novelty.

5. **`--allow-http-force-close` semantics** (§1.1). Process-startup
   flag, env var, or per-project config? Process-startup is simplest
   and matches the threat model ("an operator opted in for this
   instance"). Env var is fine too. Per-project config is overkill.
