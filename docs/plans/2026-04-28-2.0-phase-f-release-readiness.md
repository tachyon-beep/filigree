## Design decisions (locked in before tasks)

1. **Schema-mismatch is a structured `ErrorCode.SCHEMA_MISMATCH`.** The error code is already in the closed enum (`src/filigree/types/api.py:358`). All four entry points emit either:
   - **CLI / dashboard / init**: human-readable guidance to stderr, exit code 3 (distinct from generic failure exit 1).
   - **MCP**: `ErrorResponse(code=ErrorCode.SCHEMA_MISMATCH, error=<guidance text>)` returned from every `call_tool` after a schema mismatch is detected.

2. **The guidance text is shared.** A `format_schema_mismatch_guidance(installed: int, database: int) -> str` helper lives in `src/filigree/install_support/version_marker.py`. **F1 lands the module with this helper only**; F4 extends the same module with the marker read/write functions. F2/F3 import the helper from F1's module. Final wording:
   ```
   Database schema v{database} is newer than this filigree (v{installed}).
   Downgrade is not supported.

   To fix: upgrade filigree (`uv tool upgrade filigree` or your installer's
   equivalent), or use a project that was created with this version.
   Run `filigree doctor` for details.
   ```

3. **MCP "warm-but-degraded"** = server starts, `list_tools` works (so the MCP client can introspect), every other tool call returns SCHEMA_MISMATCH. This lets agents detect the condition gracefully rather than seeing a TCP/stdio reset.

4. **Exit code 3** for CLI doctor + dashboard schema mismatch. Exit 1 stays for generic CLI failures. Document this in `docs/cli.md`.

5. **`.filigree/INSTALL_VERSION` is informational, not contract.** It records the highest schema version this project has ever opened. F4 emits a warning when `init` would create a fresh DB at a version higher than the marker. Missing marker is fine — every project predating 2.0.0 won't have one. Format: single line, just the integer (e.g. `9\n`).

6. **F7 strict-xfail resolution.** The C2 `test_container_key_parity` xfail's `strict=True` says it's permanent — the work-package §F7 wants it removed. Resolution: replace the xfailed assertion with TWO positive assertions in `TestBatchMixedValidityParity`: (a) classic dashboard emits `{updated, errors}` (frozen contract), and (b) loom dashboard emits `{succeeded, failed}` (loom contract). The legacy `test_container_key_parity` test is renamed to `test_classic_container_keys_frozen` and asserts presence of the classic shape — it stops being a divergence-flagger and becomes a pin. Both surfaces' shapes are now positively asserted; zero strict-xfails remain.

7. **CHANGELOG `[2.0.0]` extends, doesn't overwrite.** The existing `[Unreleased]` entries from D7 and E7 become the body of `[2.0.0]`; F5 adds the heading, the date, the federation-framing prose, and the "Frozen" / "Stability posture" sections. Don't lose the per-phase BREAKING enumerations.

## Task 4 (F4): `filigree init` forward-warning

**Affected files:** `src/filigree/install_support/version_marker.py` (extends the module created in F1 — does NOT create it), `src/filigree/cli_commands/admin.py` (`init` command). Tests in `tests/test_schema_mismatch.py`.

**Goal:** When `filigree init` creates a fresh DB at a schema version strictly higher than any version the user has previously opened (tracked via `~/.filigree/INSTALL_VERSION`-style marker, or per-project — see step 1), emit a warning. This catches the "user upgraded filigree, runs init in an old project, doesn't realise the DB is now newer than other tools they're running."

- [ ] **Step 1: Decide the marker location**

  Two options:
  - **Per-project**: `.filigree/INSTALL_VERSION` records the version this project's DB was created or last upgraded at. Simpler; doesn't carry across projects.
  - **Per-user**: `~/.config/filigree/highest-version` or `~/.filigree/INSTALL_VERSION` records the highest version the user has ever encountered. Catches cross-project skew.

  Original design §6 change 5 specifies per-project (`.filigree/INSTALL_VERSION`). Adopt that. The forward-warning fires when:
  ```
  fresh_db_version > installed_filigree_version (impossible — we created it) → never
  fresh_db_version > previous .filigree/INSTALL_VERSION → warn
  ```
  i.e. only meaningful when the project already had a `.filigree/` (re-init / repair scenarios). For a truly fresh project with no marker, write the marker without warning.

- [ ] **Step 2: Extend `src/filigree/install_support/version_marker.py`**

  The module was created in F1 with `format_schema_mismatch_guidance()` only. F4 adds the marker constants and read/write functions to the SAME module — do NOT create a new module:

  ```python
  # Append to the existing file (created in F1):

  from pathlib import Path

  MARKER_NAME = "INSTALL_VERSION"


  def read_install_version(filigree_dir: Path) -> int | None:
      marker = filigree_dir / MARKER_NAME
      if not marker.exists():
          return None
      try:
          return int(marker.read_text().strip())
      except (OSError, ValueError):
          return None


  def write_install_version(filigree_dir: Path, version: int) -> None:
      (filigree_dir / MARKER_NAME).write_text(f"{version}\n")
  ```

  Hoist the `from pathlib import Path` to the top of the module if not already there.

- [ ] **Step 3: Wire `init`**
