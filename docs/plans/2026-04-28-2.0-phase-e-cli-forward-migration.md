# Phase E — CLI Forward-Migration to the Living Shape

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Bring the CLI surface to full parity with MCP/loom HTTP, completing ADR-002 §5's "MCP and CLI reflect the living surface only" commitment. After Phase E, every MCP tool has a CLI command that emits a loom-shape `--json` envelope and uses the same vocabulary (`issue_id`, `ListResponse[T]`, `BatchResponse[T]`, `status` not `state`).

**Architecture:** Phase E adds three new CLI modules (observations, files, scanners), introduces verb-noun aliases as permanent forms, adds `start-work`/`start-next-work` wrappers over D6's core methods, and aligns existing CLI commands with the loom envelope shapes that MCP adopted in Phase D. Each sub-task (E1–E6) is a logical commit boundary.

**Tech stack:** Click ≥ 8.1, Python 3.13. No new dependencies. The thin-wrapper pattern (CLI calls the same `FiligreeDB` methods MCP handlers call) is already established for issues/meta/planning/workflow modules — E2 extends it to the missing domains.

## Operational rules

- **Single commit boundary per sub-task** (E1, E2, … E6) — six commits, sequenced. Each commit's pre-push must be green.
- **Pre-push checklist before every commit:** `uv run ruff check src/ tests/`, `uv run ruff format --check src/ tests/`, `uv run mypy src/filigree/`, `uv run pytest --tb=short`. All four must pass; the existing strict-xfail (`tests/util/test_cross_surface_parity.py::TestBatchMixedValidityParity::test_container_key_parity`) must remain strict-xfailed.
- **Do not push, switch branches, or create branches without explicit user approval.** Branch is `2.0-project-management-extension`. The user-approval push gate is E7.
- **CLI is forward-only.** No legacy aliases that accept the old vocabulary. New verb-noun aliases (E3) are *additions* alongside existing names; they do not replace or deprecate. The loom envelope shape (`--json` output) replaces the legacy shape outright — clients pinning to the old `--json` output re-pin against the new schema.
- **Classic HTTP stays frozen.** No edits to `dashboard_routes/*.py`. The CLI calls into `FiligreeDB` core methods directly (the same path MCP uses); no HTTP round-trip.
- **AGENTS.md / CLAUDE.md will appear modified at session start** (filigree dashboard session-start hook churn). Exclude them from every commit.

## Design decisions (locked in before tasks)

1. **CLI is forward-only.** No flag like `--legacy-json` to emit the old shape. Clients pinning to legacy CLI `--json` output re-pin against the new schema. Documented prominently in the E1/E7 commit messages.

2. **Verb-noun aliases are permanent additions, not replacements.** `filigree ready` continues to work; `filigree get-ready` is added as an alias that produces identical output. Both names appear in `--help`. Per the work package and original design §4, no deprecation cycle for the short forms — they are stable.

3. **`add-label` arg order shifts to `<label> <issue_ids...>`.** Per work-package §E6 and original design §2: both `add-label` and `batch-add-label` take label first, then issues, so the variadic position is symmetric. **Breaking change for `add-label`** — the previous order was `<issue_id> <label>`. This is the only positional-arg break in Phase E; it's worth it to align with the batch sibling. Documented prominently. Existing tests update.

4. **`filigree show <issue_id>` omits files by default.** Mirrors `get_issue.include_files=False` from D4. `--with-files` opts in. `filigree files <issue_id>` (existing) is unchanged — that's the dedicated files command.

5. **`report-finding` accepts loom-shape JSON via stdin or `--file`.** Federation fallback path for emitters without HTTP clients (Clarion's `cargo run` tasks, Wardline's bash hooks). Stdin is default; `--file` overrides. Calls `db.process_scan_results(...)` directly. Returns `BatchResponse[str]` of new finding IDs (or `ScanIngestResponseLoom` if we keep the warnings/stats sibling — **decision: use ScanIngestResponseLoom for parity with the HTTP endpoint**).

6. **`start-work` / `start-next-work` argument shape mirrors the MCP tools.** Positional `<issue_id>` for `start-work`, no positional for `start-next-work`. `--assignee` required, `--target-status` optional, `--actor` optional, `--type` / `--priority-min` / `--priority-max` for `start-next-work`. Output: full IssueDict on success, `ErrorResponse` on failure.

7. **CLI list-tool `--json` output uses the loom envelope.** `filigree list --json` emits `{items, has_more, next_offset?}`, not the legacy `{issues, ...}`. Same for `search`, `ready`, `blocked`, `list-files`, `list-findings`, `list-observations`, `list-scanners`, `labels`, `events`, `changes`, `list-types`, `list-packs`. Plain (non-`--json`) output formatting is free to differ — only the `--json` envelope is contract.

8. **Container key unification on batch CLI commands' `--json` output.** `batch-update`, `batch-close`, `batch-add-label`, `batch-add-comment`, `batch-dismiss-observations`, `batch-update-findings` emit `{succeeded, failed}` per `BatchResponse[T]`. Verify `batch-close` `--json` already complies post-Phase D1 (the work-package §E1 retain-2b.3a/b/c verification step).

9. **Errors via the unified envelope.** Every CLI failure path emits `{"error": "<msg>", "code": "<UPPERCASE>"}` on `--json`, plain text on stderr otherwise, exit code 1. The `cli_common.refresh_summary()` best-effort pattern (filigree-62c5b61f68) is preserved.

## Task 4 (E4): CLI `start-work` / `start-next-work`

**Affected files:** `src/filigree/cli_commands/issues.py` (or a new `cli_commands/compose.py` if `issues.py` is over-stuffed). Tests in `tests/cli/test_workflow_commands.py` or new `tests/cli/test_compose_commands.py`.

- [ ] **Step 1: Add `start-work` command**

```python
@click.command("start-work")
@click.argument("issue_id")
@click.option("--assignee", required=True)
@click.option("--target-status", default=None)
@click.option("--actor", default=None)
@click.option("--json", "as_json", is_flag=True)
def start_work(issue_id: str, assignee: str, target_status: str | None, actor: str | None, as_json: bool) -> None:
    actor = actor or assignee
    with get_db() as db:
        try:
            issue = db.start_work(issue_id, assignee=assignee, target_status=target_status, actor=actor)
        except KeyError:
            _emit_error(f"Issue not found: {issue_id}", ErrorCode.NOT_FOUND, as_json)
            sys.exit(1)
        except (AmbiguousTransitionError, InvalidTransitionError) as exc:
            _emit_error(str(exc), ErrorCode.INVALID_TRANSITION, as_json)
            sys.exit(1)
        except ValueError as exc:
            code = classify_value_error(str(exc))
            _emit_error(str(exc), code, as_json)
            sys.exit(1)
        refresh_summary(db)
        if as_json:
            click.echo(json.dumps(issue.to_dict(), indent=2, default=str))
        else:
            click.echo(f"Started work on {issue.id}: status={issue.status}, assignee={issue.assignee}")
```

Same pattern, no positional `issue_id`. Returns the claimed issue or `{status: "empty", reason: ...}` when no match.
