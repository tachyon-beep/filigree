## 1. Why a rebaseline

Stage 2a execution was rough enough that the original 2B plan is not
trusted as-is. Three specific reasons:

1. **2b.0 scope collapsed during bed-down.** The original Stage 1 Task
   1.1 shipped a new `BatchFailure` TypedDict alongside the existing
   `BatchFailureDetail`, expecting Stage 2b.0 to do the
   `id:str / code:str → item_id:str / code:ErrorCode` migration together
   as a wire-contract change. Bed-down round 2 (commit `11cfb80`) did the
   `code` half **in place** on `BatchFailureDetail`. The wire-contract
   change is done; only the type-alias retirement remains.
2. **The parity module surfaced divergences the bed-down missed.** Four
   strict xfails and one non-shape observation (see §5) all map to 2B
   work that was not in the original plan.
3. **Clarion has no staging environment.** The user confirmed this
   explicitly. The highest-risk hop — `POST /api/v1/scan-results` shape
   or semantics changes — lands directly on production. The parity
   tests become the pre-release gate; the plan has to make that
   explicit.

The user also chose option 2c (continue local into 2B, bundle Stage 1 +
2a + bed-down + 2B into one "big 2.0 release" PR). No interim push.
This rebaseline is written against that choice.

### Task 2b.2 — Batch arg-name unification

Not surfaced by the parity module as an xfail (it is not envelope-
shape), but filed as observation `filigree-obs-92cb474341`:

- Dashboard `POST /api/batch/update` and `POST /api/batch/close` read

that passes `ids` breaks until it's updated. Rationale: the 2.0 PR
is already a big-bang release; adding a deprecation period to one arg
while 2b.1 and 2b.5 hard-cut elsewhere would be inconsistent. Run the
Clarion grep from §4 first and confirm zero live consumers before the
commit lands.

**Wire-contract impact:** Breaking for any existing MCP client that
passes `ids`. Land in the same PR as 2b.1; document both breakages
together in the CHANGELOG.

### Task 2b.3 — CLI Click-layer envelope coverage

Three related xfails, all from the parity module:

- `TestPriorityOutOfRangeCreateParity::test_cli_emits_envelope`
  (`cli_commands/issues.py:23` — `click.IntRange(0, 4)` on
  `--priority`).
- `TestBlankActorUpdateParity::test_cli_emits_envelope` (`cli.py:25`
  — group-level `--actor` validator raises `click.BadParameter`).
- `TestAlreadyClosedParity::test_cli_emits_flat_envelope`
  (`cli_commands/issues.py:279ish` — `close <id> --json` always emits
  batch-shape wrapper, not flat envelope, even for N=1).

All three share a root cause: Click validates before the command body
runs (or before the `--json` flag is visible at the group level).

**Chosen approach (user decision, 2026-04-23):** per-option callbacks.
`IntRange(0, 4)` → `callback=_priority_or_envelope(ctx, value)` that
inspects `ctx.params` for `--json` (or detects the flag via
`sys.argv`) and emits JSON envelope + `sys.exit(1)` if set, else
re-raises `click.BadParameter`. Localises the fix per option/command
and keeps the CLI lint-clean and Click-idiomatic at the schema level.

Three call sites to migrate, all in the CLI package:

- `cli_commands/issues.py:23` — `--priority` on `create`.
- `cli.py:25` — group-level `--actor` validator (this is the hardest
  because the group runs before subcommand `--json` is parsed — may
  need to fall back to a `sys.argv` sniff for the `--json` flag at
  this one site; document the sniff in a comment).
- `cli_commands/issues.py` `close()` (around line 279) — when called
  with a single issue id, emit flat envelope instead of the batch
  wrapper.

The group-level shim (previous option 2) and accept-and-delete
(previous option 3) are off the table.

For the `close <id> --json` N=1 wrapper: emit flat envelope when
`len(issue_ids) == 1` fails at the single-id path. Keep batch-shape for
N≥2.

**Wire-contract impact:** CLI-only, additive for `--json` paths (new
envelope shape where nothing was emitted before, or flat envelope
replacing batch-wrapper for N=1 close).

### Task 2b.4 — `classify_value_error` boundary rule (docs + lightweight enforcement)

The bed-down left a live heuristic in `src/filigree/types/api.py`
(`classify_value_error`) used at ~7 sites. Without a documented
boundary rule it will regress: someone will apply it at an input-
validation site and mis-classify validation-class errors as something
else.

**Rule to document in the `classify_value_error` docstring and in
per-site comments at enforcement sites:**

- **State-machine sites use the helper.** Current call sites:
  `db._batch_with_transition_errors`, `_handle_update_issue`,
  `_handle_close_issue`, `_handle_reopen_issue`, CLI `update`/`close`/
  `reopen` `--json` paths. These wrap a `db.*` method that raises
  `ValueError` for multiple classes of issue: transition failure,
  "already closed", dependency conflict, validation. The helper routes
  correctly across these.
- **Input-validation sites stay hardcoded `VALIDATION`.** Confirmed
  sites: `_safe_path` and URL parsing in
  `dashboard_routes/common.py`; schema validation in
  `mcp_tools/meta.py`, `mcp_tools/files.py`, `mcp_tools/scanners.py`,
  `mcp_tools/observations.py`. These catch `ValueError` for one class
  only (bad input); using the helper here would mis-classify future
  error-message additions.

**Lightweight enforcement (optional but cheap):** a one-line test in
`tests/util/` that greps the input-validation modules listed above
for `classify_value_error(` imports and fails if any appear. Catches
future drift where someone reaches for the helper at a site it
shouldn't be used at.

```python
def test_classify_value_error_not_used_at_input_validation_sites() -> None:
    INPUT_VALIDATION_MODULES = [
        "src/filigree/dashboard_routes/common.py",  # _safe_path, URL parsing
        "src/filigree/mcp_tools/meta.py",
        "src/filigree/mcp_tools/files.py",
        "src/filigree/mcp_tools/scanners.py",
        "src/filigree/mcp_tools/observations.py",
    ]
    for path in INPUT_VALIDATION_MODULES:
        content = Path(path).read_text()
        assert "classify_value_error" not in content, (
            f"{path} is an input-validation site; it must hardcode "
            f"ErrorCode.VALIDATION, not route through classify_value_error. "
            f"See docs/plans/2026-04-23-2.0-stage-2b-rebaseline.md §Task 2b.4."
        )
```

**Wire-contract impact:** None (docs + comments + one test). Fold
into the PR alongside the code changes.

### Task 2b.5 (original plan) — `ListResponse[T]` rollout

From the original plan §Stage 2b: every list endpoint (MCP +
dashboard) returns `ListResponse[T]` with `next_cursor` and
`has_more` metadata.

**This is a breaking wire-contract change by construction.** A
top-level JSON array cannot gain sibling metadata keys — you either
keep it a list (status quo) or wrap it as
`{items:[...], next_cursor, has_more}` (breaking). The "additive"
framing is incoherent; there is no version of this that doesn't break
existing array-consuming clients.

**Chosen approach (user decision, 2026-04-23): strict-wrap.**
`ListResponse[T]` is `{items:[...], next_cursor, has_more}` on every
list endpoint. Land alongside 2b.1 / 2b.2 in the `### Changed` section
of the CHANGELOG. Descope is off the table — the "clean surface"
promise loses value if pagination metadata is missing.

**Wire-contract impact:** Breaking for every list-returning endpoint
on both surfaces. Clarion grep (see §4) extends here too: before
landing, `rg -n 'list_issues\|list_files\|list_scanners\|list_observations\|list_labels' <clarion>/`
and record the hits.

## 4. Risks & gates

### Clarion dogfood — no staging environment

User confirmed there is no Clarion staging env. The highest-risk hop
in 2B is the Clarion-facing `POST /api/v1/scan-results`, and any 2b.1
container-key change that touches it lands directly on production.

**Mitigation — the parity module is the pre-release gate.** Four tests
in `tests/util/test_cross_surface_parity.py::TestScanResultsEnvelope`
currently pin the **error paths**:

- Missing `findings` → 400 + flat envelope + `VALIDATION`
- Non-list `findings` → 400 + flat envelope + `VALIDATION`
- Missing `scan_source` → 400 + flat envelope + `VALIDATION`
- Non-JSON body → 400 + flat envelope

**Known gap: the 200 success shape is not pinned.** `POST /api/v1/scan-results`
returns whatever `db.process_scan_results(...)` produces (dashboard_routes/
files.py:329-339). If any 2B task touches that return dict, nothing in the
parity module catches the break. This is a real hole in "the clean surface"
promise.

**Close this gap BEFORE 2b.1 starts** as task 2b.-1 (see §3). One commit,
same structure as the error tests: call `POST /api/v1/scan-results` with
a valid empty-findings body, assert the returned dict's key set and the
types of the values. That becomes the success-path gate.

Any 2B shape change that **breaks any of these invariants** (error
envelopes or success shape) is itself a breaking change and must:

1. Update the parity tests in the same commit.
2. Appear in `CHANGELOG.md [Unreleased] ### Changed` with the specific
   shape delta.
3. Be verified against Clarion's current source (see next subsection)
   before landing.
