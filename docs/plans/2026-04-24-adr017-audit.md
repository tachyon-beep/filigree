# ADR-017 Semantics Audit — 2026-04-24

**Context**: Phase A3 of the [2.0 federation work package](./2026-04-24-2.0-federation-work-package.md). Before pinning classic contract fixtures (Phase A2), confirm the 2.0 branch has not inadvertently drifted from the `POST /api/v1/scan-results` semantics that Clarion's ADR-017 relies on. A fixture that freezes a drifted shape would anchor Clarion's future pin to the wrong behaviour.

**Source of truth**: `/home/john/clarion/docs/clarion/adr/ADR-017-severity-and-dedup.md`.

**Audit scope**: the three ADR-017 load-bearing semantics named in the work package — `mark_unseen` default, `scan_run_id` resume behaviour, and dedup key — plus a brief check on `complete_scan_run` which governs whether a scan run is finalized at the end of an ingest.

## Findings

### 1. `mark_unseen` default — UNCHANGED

Handler: `src/filigree/dashboard_routes/files.py:315` — `mark_unseen = body.get("mark_unseen", False)`.

Core: `src/filigree/db_files.py:624` — `mark_unseen: bool = False`.

ADR-017 expectation (§Dedup policy): Clarion explicitly passes `mark_unseen=true` on every batch. The default must remain `False` so non-Clarion callers do not accidentally sweep findings into `unseen_in_latest` without opting in.

**Result**: default `False` preserved in both layers. Clean.

### 2. `scan_run_id` resume semantics — UNCHANGED

`_update_existing_finding` (`db_files.py:567+`):

- On re-matched finding (same dedup tuple), status is raised from terminal back to open: `status = CASE WHEN status IN ('fixed', 'unseen_in_latest') THEN 'open' ELSE status END`.
- `scan_run_id` follows first-attribution-wins: `scan_run_id_update = scan_run_id if (scan_run_id and not existing_run_id) else existing_run_id` (db_files.py:574).
- Re-matched findings are NEVER transitioned to `unseen_in_latest` on resume. That transition happens only via `_mark_unseen_findings`, which runs only when the caller passes `mark_unseen=True` (db_files.py:689).

`complete_scan_run` semantics (`db_files.py:703`):

- Default: `True`. On completion, the run is finalized via `update_scan_run_status(scan_run_id, "completed", ...)`.
- When an orchestrator wants to reuse a `scan_run_id` across multiple batches, it passes `complete_scan_run=False` on intermediate batches and `complete_scan_run=True` on the final one. ADR-017 §Dedup policy describes exactly this pattern.

**Result**: resume semantics match ADR-017. Clean.

### 3. Dedup key — UNCHANGED vs ADR-017 (note: work-package text drifted from reality)

`db_files.py:486-490` issues the dedup SELECT:

```sql
SELECT id, seen_count, scan_run_id, issue_id FROM scan_findings
 WHERE file_id = ?
   AND scan_source = ?
   AND rule_id = ?
   AND coalesce(line_start, -1) = ?
```

Schema index: `db_schema.py:158` — `CREATE INDEX ... ON scan_findings(scan_run_id)` is the run-lookup index. The dedup uniqueness (enforced at the application layer via the SELECT-then-UPDATE-or-INSERT path) matches the 4-tuple above.

ADR-017 §Dedup policy line 80: _"Filigree's server-side dedup key is `(file_id, scan_source, rule_id, coalesce(line_start, -1))`."_ This is the 4-tuple the code uses.

**Note on internal doc drift**: the work package at `docs/plans/2026-04-24-2.0-federation-work-package.md:82` summarises the dedup key as "`(rule_id, file_path, line_start)` tuple" — a loose 3-tuple. This is a simplification in the work package, not a code drift. The audit here records the real 4-tuple so the Phase A2 fixture reasoning is anchored to the ADR-017 wording, not the work-package wording. No code change needed; the work package text can be corrected in a later pass if desired (low priority).

**Result**: code and schema match ADR-017. Clean. One doc-text simplification noted.

## Verdict

**Audit clean.** All three ADR-017 load-bearing semantics are preserved on the 2.0 branch. Phase A2 may proceed; the classic fixture will pin the current shape without freezing drift.
