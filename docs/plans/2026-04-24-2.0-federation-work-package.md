## READ THIS FIRST (entry point for the executing agent)

You are about to pick up work on Filigree 2.0. The plan has gone through three iterations (2026-04-18 design, 2026-04-23 rebaseline, 2026-04-24 framing shift). This document supersedes the execution instructions in the prior plans. The prior plans remain useful as **shape references** — what the envelopes, error codes, and vocabulary should look like — but **not as execution sequences**.

**Your north star is ADR-002** (`docs/architecture/decisions/ADR-002-api-generations-and-federation-posture.md`). Read it before touching code. It defines *what we are building and why*. This document defines *how and in what order*.

**The core reframing:** Filigree 2.0 is not a break-and-bundle release. It is an **additive** release that introduces a named API generation (`loom`) alongside the frozen classic generation (`/api/v1/*`). MCP and CLI forward-migrate to the living / loom shape. Named generations have lifecycles decoupled from code versions.

### Phase B — Generation infrastructure

**B1. Routing split: classic-at-current-URLs and `/api/loom/*`.**

> **2026-04-24 amendment (reality-aligned).** The original text here said "`classic_router` mounted at `/api/v1`". That assumed all classic URLs lived under `/v1/`, but only `POST /api/v1/scan-results` actually does. Every other existing endpoint is unversioned (`/api/issues`, `/api/files`, `/api/stats`, etc.). Mounting classic_router at `/api/v1` uniformly would force a URL move on every existing consumer — the opposite of ADR-002's "classic is frozen" principle. The amended plan below keeps classic URLs where they are and only adds loom at a new path.

Introduce two APIRouters in the project-router factory (`_create_project_router` in `src/filigree/dashboard.py`):

- `classic_router` — carries every currently-existing project-scoped endpoint at its current path. Most routes are unprefixed (e.g. `/issues`, `/files`, `/stats`, `/ready`) and become `/api/<path>` after the app-level prefix. The one outlier is `POST /v1/scan-results`, which becomes `/api/v1/scan-results`. No URL moves.
- `loom_router` — new, empty in Phase B, mounted at `/api/loom` via the project-router factory. Phase C fills it.

The `app.include_router(router, prefix="/api")` mount (and the server-mode `/api/p/{project_key}` mount) applies to the project-router factory's output, which includes **both** classic_router and loom_router. So server-mode consumers get `/api/p/{key}/issues` (classic) and `/api/p/{key}/loom/...` (loom) for free from the same wiring.

MCP streamable-HTTP (`/mcp` mount in `dashboard.py`) and infrastructure endpoints (`/`, `/api/health`, `/api/projects`, `/api/reload`, `/static`) are out of scope — MCP reflects the living surface per ADR-002 §5, and the infrastructure endpoints are app-scoped, not generation-scoped.

Phase B restructure:

- `dashboard_routes/issues.py`, `files.py`, `analytics.py`, `releases.py` — each currently exports `create_router()`. Keep that as the classic factory (rename to `create_classic_router()` for clarity, with a backward-compatible `create_router = create_classic_router` alias to avoid churning unrelated imports during B1) and add a parallel `create_loom_router()` returning an empty `APIRouter` for Phase B.
- `_create_project_router()` composes the two generations into the same per-project mount tree: classic routes at their existing paths, loom routes under a `/loom/` sub-prefix on the same project router (so both sub-routers land under `/api/` and `/api/p/{key}/`).

**B2. Shape adapter pattern.**

Create `src/filigree/generations/__init__.py` and `src/filigree/generations/loom/adapters.py`, `src/filigree/generations/classic/adapters.py`.

Adapters take internal domain objects (or internal-shape dicts using `issue_id` vocabulary) and produce generation-appropriate wire shapes:

```python
# src/filigree/generations/classic/adapters.py
def slim_issue_to_classic(issue: Issue) -> dict[str, Any]:
    """Convert internal Issue to classic wire shape with 'id' key."""
    return {
        "id": issue.id,
        "title": issue.title,
        "status": issue.status,
        "priority": issue.priority,
        "type": issue.type,
    }

# src/filigree/generations/loom/adapters.py
def slim_issue_to_loom(issue: Issue) -> SlimIssueLoom:
    """Convert internal Issue to loom wire shape with 'issue_id' key."""
    return SlimIssueLoom(
        issue_id=issue.id,
        title=issue.title,
        status=issue.status,
        priority=issue.priority,
        type=issue.type,
    )
```

Define loom-specific TypedDicts in `src/filigree/generations/loom/types.py` (`SlimIssueLoom`, `BatchResponseLoom[T]`, etc.). Classic uses the existing types in `src/filigree/types/api.py` unchanged.

**B3. Living surface routing — deferred to Phase C.**

> **2026-04-24 amendment.** The original text here prescribed mounting a third router at `/api/*` aliasing loom. That runs into a collision with the reality discovered during the B1 amendment: most existing `/api/<path>` URLs are already classic (frozen), so a living-surface alias at the same path would conflict with the classic route on the same path. The ADR-002 §2 principle ("`/api/*` aliases the current recommended generation") remains correct; it just cannot be implemented as a blanket mount on top of the existing classic layout.

Deferred to Phase C. The resolution will be made per-endpoint when a concrete loom endpoint forces the question:

- For endpoints that exist in classic under `/api/<path>` and are being *added* to loom under `/api/loom/<path>`, the living-surface alias at `/api/<path>` is ambiguous — it is already bound to classic. Two options surface at that point:
  1. Treat the existing `/api/<path>` as *both* classic and living (they coincide for that endpoint until a future generation redefines the shape). When loom diverges, the living-surface alias moves to loom.
  2. Reserve `/api/<path>` for classic permanently, and have the living surface use a different unversioned convention (e.g. `/api/current/<path>`) for disambiguation.
- For endpoints that are *new* in loom with no classic counterpart (e.g. new composed operations like `/api/loom/issues/{id}/start-work`), the living-surface alias at `/api/<path>` is unambiguous and can be added per the ADR §2 rule.

Phase C will pick per-case and document the precedent. For Phase B, living surface is a forward-declared concept, not a mounted route. `/api/<path>` for existing endpoints remains classic-only until Phase C's per-endpoint resolution.

Per ADR §2, every living-surface endpoint *when it exists* requires a docstring declaring `"Equivalent to /api/loom/<path> as of <date>."` — adopt in Phase C at the point of mounting.

**B4. Parity-per-generation test infrastructure.**

Extend `tests/util/test_cross_surface_parity.py` with generation-aware test classes:

- `TestClassicGenerationParity` — for each classic endpoint present in a contract fixture, assert the live response matches the fixture's shape contract (key set + value types + enum closure + status/envelope pairing per `docs/federation/contracts.md`). In Phase B the only classic fixture is `scan-results.json`; more join as fixtures are published.
- `TestLoomGenerationParity` — for each loom endpoint present in a contract fixture, assert the same. Phase B classes land as placeholders that are `pytest.mark.skip` until Phase C mounts the loom endpoints. Converting skip → pass is the Phase C gate per-endpoint.
- `TestLivingSurfaceEquivalence` — **deferred to Phase C** alongside B3. When a living-surface route is mounted, this class asserts its response is shape-equivalent (not byte-equivalent — per `docs/federation/contracts.md`, filigree does not guarantee byte-determinism) to the corresponding named-generation route on the same inputs.

These gate the correctness of the dual-surface model.

**Gate before Phase C:** `pytest tests/util/test_cross_surface_parity.py` green against empty loom router. Classic fixture pins pass. Shape adapter modules importable and mypy-clean.

### What will feel strange if you come from the earlier plans

The new reason is "we're still bundling for review convenience, but no consumer depends on the bundling because Clarion pins to loom and classic is unchanged." - **"Stage 6 renames last" is still true, but less load-bearing.** Stage 6 is folded into Phase D (MCP forward-migration). The concern about "namespace shift forcing earlier stages to rebase" is dissolved because classic generation preserves the pre-rename wire and loom is new greenfield.
- **Parity module xfails now map to Phase C / D landings, not to Stage 2B.** The container-key xfail converts to passing when Phase C2 lands (loom batch endpoints). Do not delete xfails prematurely; they are the gate.

### What is easy that was hard

- **Landing order inside a phase is flexible.** Phase C's five tasks can interleave; only the phase→phase gates are strict.
- **Rollback is per-commit, not per-phase.** Each loom endpoint addition is independently revertable. If a loom endpoint lands and reveals a problem, it can be reverted while classic stays untouched.
- **Contract fixtures catch regressions at CI time, not production time.** If a classic adapter drifts, parity tests fail. This replaces the "Clarion has no staging" anxiety with a local gate.

### What is hard that was easy (or previously deferred)

- **Two surfaces means two sets of documentation per endpoint.** Phase F's doc update is larger than the original plan's estimate. Budget accordingly; do not skip classic documentation on the assumption it is unchanged (it is unchanged in shape, but a consumer reading it for the first time needs it to say so).
- **Shape adapters must be thin.** Resist the temptation to put behavior in adapters. If you find yourself writing an `if generation == "classic": ...` branch, the logic belongs in the handler, not the adapter. Adapters rename keys and wrap envelopes; they do not make decisions.
- **Living surface documentation freshness** requires per-endpoint discipline. Every living-surface handler needs a docstring declaring its generation-of-the-day; CI enforces presence, but accuracy is on the author.
