## 1. Release theme

2.1.0 closes two work streams:

1. **Cross-product entity binding (ADR-029).** Already merged on
   `clarion/b7-entity-associations` (PRs #41 reverse-lookup work and #42
   review nits). New `entity_associations` table (schema v15), four MCP
   tools, matching HTTP routes. Currently sitting in `[Unreleased]`.
   The v14→v15 schema bump alone obligates a minor-version cut — there
   is no clean 2.0.x cut that includes this work.
2. **`db_issues.py` hardening** per the 2026-05-17 design — optimistic-lock
   correctness, audit-trail integrity, HTTP attack-surface trim, SQLite
   transaction discipline, and backward edges as first-class template
   concepts. Removes `_skip_transition_check` (the design's "shifting
   the burden" archetype) as the closing act.

Because (1) already forces a minor bump, the design doc's "Should we ship
Phase 0 as 2.0.5?" open question collapses: everything ships as 2.1.0.

## 2. Scope decisions (locked pending user confirmation)

These resolve the design doc's open questions so execution doesn't
relitigate them. The first phase PR is the implicit ratification gate —
flag any disagreement before that PR opens.

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | All db_issues hardening phases ship in **2.1.0** (no 2.0.5 split). | Schema bump from entity-associations already obligates a minor cut; one coherent release-note story. |
| 2 | **`event_seq INTEGER` column** for §0.2, not millisecond `created_at`. | Clock-independent; design doc preference; small forward-only migration. |

| 6 | `WrongProjectError.safe_message` mirrors `ForeignDatabaseError.safe_message` exactly. | Established pattern from PR #41; symmetric redaction. |
| 7 | Actor verification stays **out of scope** (design §1.4); ADR + length cap only. | Real fix requires session-token mechanism — separate work package. |
| 8 | Drop the design doc's planned separate implementation plan (`2026-05-NN-2.1-db-issues-hardening-plan.md`). This release-prep doc subsumes it. | One artifact, not two. |

## 3. Release sequence and gates

Two gates: **alpha cut** (Phase 0 + entity-associations + entry sanity)
and **final cut** (everything). Each phase below is a commit boundary
unless a finding is too small to stand alone. PR per phase is the
recommended granularity.

### Branching

- The 2.1.0 work needs a release branch separate from `main`.
  **Recommended:** create `release/2.1.0` off `main` after the
  current `clarion/b7-entity-associations` branch merges. **Do not

```bash
uv run ruff check src/ tests/

uv run mypy src/filigree/
```

All four pass. No `--no-verify`. Main has branch protection — PR review
required.

### Phase 0 — Critical correctness

Lands first; provides typed exceptions that Phase 1 batch handlers depend
on, and establishes the IMMEDIATE-tx position Phase 2 will generalise.

- [ ] **§0.1 — `update_issue` atomic CAS on assignee** (`src/filigree/db_issues.py:638–880`)
  - [ ] Add `AND assignee = ?` to UPDATE when an assignee was observed.
  - [ ] On `rowcount == 0`, re-read and raise `ClaimConflictError` (defined in §0.3 — land §0.3 first or co-land).
  - [ ] Mirror to `close_issue`, `batch_close`, `batch_update`.
  - [ ] Tests: `test_update_issue_cas_rejects_concurrent_reassign`,
        `test_close_issue_inherits_cas`,
        `test_batch_close_per_item_cas`.

- [ ] **§0.3 — Typed `ClaimConflictError`** (`src/filigree/db_base.py` or `src/filigree/types/api.py`)
  - [ ] Define `class ClaimConflictError(ValueError)`.
  - [ ] `_check_expected_assignee` raises the new class.
  - [ ] CAS-failure paths in `claim_issue` / `release_claim` / `reclaim_issue` / `heartbeat_work` raise the new class.
  - [ ] Replace the five string-matched dispatch sites (`db_issues.py:1290, 1688, 1793, 1830, 1865`) with `isinstance` checks; `ErrorCode.CONFLICT` for `ClaimConflictError`, `VALIDATION` for bare `ValueError`.
  - [ ] Tests: `test_claim_conflict_raises_typed_exception`,
        `test_batch_close_reports_conflict_for_claim_conflict`.

- [ ] **§0.4 — `WrongProjectError` precedence in batch handlers**
  - [ ] Every batch handler (`batch_close`, `batch_update`, `batch_add_label`, `batch_remove_label`, `batch_add_comment`, `release_my_claims`, plus `_batch_with_transition_errors`) catches `WrongProjectError` and `ClaimConflictError` before generic `ValueError`.
  - [ ] `WrongProjectError` aborts the whole batch with envelope-level error, not per-item.
  - [ ] Tests: `test_batch_close_foreign_prefix_aborts_batch_not_per_item`,
        `test_release_my_claims_foreign_prefix_aborts`.

- [ ] **§0.2 — `_record_event` collision visibility** (`src/filigree/db_events.py:96–100`, schema `src/filigree/db_schema.py:64–66`, `src/filigree/migrations.py`)
  - [ ] Add migration v15→v16: new column `event_seq INTEGER NOT NULL DEFAULT 0`; rebuild UNIQUE index over `(issue_id, event_type, actor, old_value, new_value, created_at, event_seq)`.
  - [ ] `_record_event` switches from `INSERT OR IGNORE` to plain `INSERT`. Compute `event_seq` inline via a per-issue subquery in the same INSERT statement:
        ```sql
        INSERT INTO events (issue_id, event_type, ..., event_seq)
        VALUES (?, ?, ..., COALESCE((SELECT MAX(event_seq) FROM events WHERE issue_id = ?), -1) + 1)
        ```
        This is multi-process-safe (no in-memory counter), survives crashes, and runs inside the IMMEDIATE tx so the SELECT-then-INSERT is atomic from any other writer's perspective. Adds one indexed lookup per event write — acceptable.
  - [ ] `IntegrityError` on true duplicates bubbles up to caller for transaction rollback.
  - [ ] Tests: `test_heartbeat_bursts_record_every_event`,
        `test_record_event_sequence_monotonic`,
        `test_record_event_dedup_still_fires_on_true_duplicate`.
  - [ ] Migration test in `tests/install/test_migrate.py` (or local equivalent): historical rows survive, all default `event_seq = 0`.

**Phase 0 commit boundaries:** three PRs is reasonable — (§0.3 + §0.4) as
the typed-exception PR, §0.1 as the CAS PR (depends on §0.3), §0.2 as the
schema PR. Or one bundled PR if the test corpus is small enough.

## 8. Rollback & post-release monitoring

- **Schema rollback is not supported** (per the 2.0.4 ADR). If
  catastrophic, the only path is point-in-time DB restore.
- **CHANGELOG entry includes the rollback story** so users with mixed
  agent versions understand the no-downgrade boundary.
- **Post-release: watch for** any reports of `IntegrityError` from
  `_record_event` (legitimate true-duplicate caller bugs that the old
  silent-ignore was masking). If they surface in user reports, classify
  per-bug rather than reverting the integrity guarantee.
