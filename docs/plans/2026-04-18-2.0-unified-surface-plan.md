# Filigree 2.0 Unified Surface Implementation Plan

> **Execution sequencing superseded 2026-04-24.** The stage-by-stage break-and-bundle execution defined below is superseded by the phased dual-surface model in [ADR-002](../architecture/decisions/ADR-002-api-generations-and-federation-posture.md) and the [2.0 federation work package](./2026-04-24-2.0-federation-work-package.md). Specific task content here (shape references, fixture patterns, test scaffolds) remains useful as a shape reference; the ordering and landing strategy do not. Do not execute from this plan.

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship Filigree 2.0 — a coherent MCP+CLI surface with a unified
response envelope (`BatchResponse` / `ListResponse`), closed `ErrorCode`
enum, composed `start_work` operation, full CLI parity for
observations/files/scanners, compact `get_issue` default, typed
schema-mismatch UX, and normalized `issue_id` / `status` vocabulary.

**Architecture:** Seven sequential stages on the
`2.0-project-management-extension` branch, each its own PR. Stage 1 adds
types; 2a/2b roll out the envelope in two halves; a dogfood gate catches
shape drift; 3–5 add features additively; 6 is the breaking-rename sweep;
7 ships release docs.

**Tech Stack:** Python 3.11+ (per `pyproject.toml`; `StrEnum` requires 3.11),
TypedDict/StrEnum/Generic from `typing`/`enum`, Click for CLI, MCP Python SDK,
SQLite via internal `FiligreeDB` mixin composition, `uv` for tool management,
`ruff` + `mypy` + `pytest` (+ `-x --tb=short`) for gates.

**Source spec:** `docs/plans/2026-04-18-2.0-unified-surface-design.md`
**Source review:** `docs/plans/2026-04-18-mcp-cli-interface-review.md`

```python
class SchemaVersionMismatchError(Exception):
    """Raised when installed filigree is older than the project's DB schema.

    Carries typed fields so callers can pattern-match on it without
    string-matching the message.
    """

    def __init__(self, *, installed: int, database: int) -> None:
        self.installed = installed
        self.database = database
        super().__init__(
            f"Database schema v{database} is newer than this version of "
            f"filigree (expects v{installed}). Downgrade is not supported."
        )
```

```python
# ---------------------------------------------------------------------------
# 2.0 response envelopes

from typing import Generic, TypeVar

_T = TypeVar("_T")


class BatchFailure(TypedDict):
    """One failed item inside a BatchResponse.failed list.

    ``item_id`` is deliberately generic — batch operations exist for
    issues, findings, and observations; this field carries whichever id
    shape the specific batch tool operates on.
    """

    item_id: str
    error: str
    code: "ErrorCode"


class BatchResponse(TypedDict, Generic[_T]):
    """Unified response for batch mutation operations.

    Rules:
    - ``succeeded`` is a list of T — SlimIssue for issue-centric ops in
      slim mode, full records for others or in full mode.
    - ``failed`` is always present (empty list if no failures).
    - ``newly_unblocked`` is OMITTED entirely when the op cannot unblock
      (not present as ``[]``). Present only for close/transition ops.
    """

    succeeded: list[_T]
    failed: list[BatchFailure]
    newly_unblocked: NotRequired[list["SlimIssue"]]


class ListResponse(TypedDict, Generic[_T]):
    """Unified response for list/query operations.

    - ``items`` is always present (empty list for no results).
    - ``has_more`` is always present (never omitted); callers can
      reliably distinguish "no more" from "field absent."
    - ``next_offset`` is present only when ``has_more=True``.
    """

    items: list[_T]
    has_more: bool
    next_offset: NotRequired[int]
```

If `SlimIssue` is defined below the new block, use a string forward-ref
`"SlimIssue"`; otherwise use the direct type. Verify with the existing

```python
# ---------------------------------------------------------------------------
# 2.0 typed exceptions
# ---------------------------------------------------------------------------


class SchemaVersionMismatchError(Exception):
    """Raised when installed filigree is older than the project's DB schema.

    Both ``filigree doctor`` and the dashboard/mcp_server catch this by
    type, so callers can pattern-match on it without string-matching the
    message.
    """

    def __init__(self, *, installed: int, database: int) -> None:
        self.installed = installed
        self.database = database
        super().__init__(
            f"Database schema v{database} is newer than this version of "
            f"filigree (expects v{installed}). Downgrade is not supported."
        )


class AmbiguousTransitionError(Exception):
    """Raised by WorkflowPack.canonical_working_status when multiple
    wip-category targets exist from the current status."""

    def __init__(self, type_name: str, candidates: list[str]) -> None:
        self.type_name = type_name
        self.candidates = candidates
        super().__init__(
            f"start_work ambiguous for type {type_name!r}: "
            f"multiple wip-category targets available ({candidates}). "
            f"Specify target_status explicitly."
        )


class InvalidTransitionError(Exception):
    """Raised by WorkflowPack.canonical_working_status when no wip-category
    target is reachable from the current status."""

    def __init__(self, type_name: str, current_status: str) -> None:
        self.type_name = type_name
        self.current_status = current_status
        super().__init__(
            f"No wip-category transition from {current_status!r} for type "
            f"{type_name!r}."
        )
```

```python
    object with `.content[0].text` (a JSON string).

    def __init__(self, db: FiligreeDB) -> None:

        from filigree import mcp_server as _mod

        # mcp_server.py (verified 2026-04-18) exposes a module-level
```

```python
def _handle_batch_close(self, args: BatchCloseArgs) -> list[TextContent]:
    ids = args.get("ids", [])
    if not isinstance(ids, list) or not all(isinstance(i, str) for i in ids):
        return _text(ErrorResponse(
            error="'ids' must be a list of strings",
            code=ErrorCode.VALIDATION,
        ))

    succeeded: list[SlimIssue] = []
    failed: list[BatchFailure] = []
    newly_unblocked: list[SlimIssue] = []

    for issue_id in ids:
        try:
            result = self.db.close_issue(issue_id, reason=args.get("reason"))
            succeeded.append(_slim(result["issue"]))
            for unb in result.get("unblocked", []):

        except IssueNotFoundError as e:

        except Exception as e:
            failed.append({"item_id": issue_id, "error": str(e), "code": ErrorCode.IO})

    response: BatchResponse[SlimIssue] = {"succeeded": succeeded, "failed": failed}
    if newly_unblocked:
        response["newly_unblocked"] = newly_unblocked
    return _text(response)
```

```python
def _claim_issue_nocommit(
    self,
    issue_id: str,
    *,
    assignee: str,
    actor: str = "",
) -> Issue:
    """Claim-logic body without commit/rollback.

    Does NOT call `self.conn.commit()` / `self.conn.rollback()`. The
    caller must wrap this in a transaction boundary (e.g.,
    `with self.conn:` or by handling commit/rollback around
    multi-step operations). Used by both `claim_issue` (for the
    solo-op path) and `start_work` (composed with a status transition).

    Raises:
        IssueNotFoundError: issue_id is not in the DB.
        ClaimConflictError: another assignee holds the claim.
        ValueError: validation failure (empty assignee, wrong status).
    """
    from filigree.types.api import ClaimConflictError, IssueNotFoundError

    if not assignee or not assignee.strip():
        msg = "Assignee cannot be empty"
        raise ValueError(msg)
    self._check_id_prefix(issue_id)

    row = self.conn.execute(
        "SELECT type, assignee FROM issues WHERE id = ?", (issue_id,)
    ).fetchone()
    if row is None:
        raise IssueNotFoundError(issue_id)
    issue_type = row["type"]
    old_assignee = row["assignee"] or ""

    open_states: list[str] = []
    tpl = self.templates.get_type(issue_type)
    if tpl is not None:
        open_states = [s.name for s in tpl.states if s.category == "open"]
    if not open_states:
        open_states = ["open"]

    status_ph = ",".join("?" * len(open_states))
    cursor = self.conn.execute(
        f"UPDATE issues SET assignee = ?, updated_at = ? "
        f"WHERE id = ? AND status IN ({status_ph}) "
        f"AND (assignee = '' OR assignee IS NULL OR assignee = ?)",
        [assignee, _now_iso(), issue_id, *open_states, assignee],
    )

    if cursor.rowcount == 0:
        current = self.conn.execute(
            "SELECT status, assignee FROM issues WHERE id = ?", (issue_id,)
        ).fetchone()
        if current is None:
            raise IssueNotFoundError(issue_id)
        if current["assignee"] and current["assignee"] != assignee:
            raise ClaimConflictError(issue_id, current_assignee=current["assignee"])
        msg = (
            f"Cannot claim {issue_id}: status is {current['status']!r}, "
            f"expected open-category state"
        )
        raise ValueError(msg)

    self._record_event(
        issue_id, "claimed", actor=actor,
        old_value=old_assignee, new_value=assignee,
    )
    return self.get_issue(issue_id)


def claim_issue(
    self,
    issue_id: str,
    *,
    assignee: str,
    actor: str = "",
) -> Issue:
    """Atomically claim an open-category issue with optimistic locking.

    Public entry point — owns its own transaction boundary. See
    `_claim_issue_nocommit` for composable use inside `start_work`.
    """
    try:
        issue = self._claim_issue_nocommit(issue_id, assignee=assignee, actor=actor)
        self.conn.commit()
    except Exception:
        self.conn.rollback()
        raise
    return issue
```

```python
def canonical_working_status(
    self, type_name: str, current_status: str
) -> str:
    """Return the unique `wip`-category status reachable from
    current_status for type_name.

    Raises:
        AmbiguousTransitionError: multiple wip-category targets exist.
        InvalidTransitionError: no wip-category target exists OR
            current_status is not a state in this type's workflow.
        KeyError: the type_name is not defined in this pack (caller bug).
    """
    from filigree.types.api import (
        AmbiguousTransitionError,
        InvalidTransitionError,
    )

    tpl = self.types[type_name]   # KeyError propagates — caller bug

    # Build a name→StateDefinition lookup since `states` is a tuple.
    by_name = {s.name: s for s in tpl.states}
    if current_status not in by_name:
        raise InvalidTransitionError(type_name, current_status)

    # Transitions live on TypeTemplate, not StateDefinition.
    wip_targets: list[str] = []
    for transition in tpl.transitions:
        if transition.from_state != current_status:
            continue
        target = by_name.get(transition.to_state)
        if target is not None and target.category == "wip":
            wip_targets.append(transition.to_state)

    if len(wip_targets) == 0:
        raise InvalidTransitionError(type_name, current_status)
    if len(wip_targets) > 1:
        raise AmbiguousTransitionError(type_name, wip_targets)
    return wip_targets[0]
```

```python
def test_start_work_atomicity_two_callers(mcp_client_with_open_bug) -> None:
    """Concurrent start_work from two callers: exactly one wins, AND the
    loser's status did NOT advance (the atomicity invariant).

    This is the central guarantee of start_work — the transition must
    only land for the winner. If the transition leaked for the loser,
    we'd see two "confirmed" bugs with one of them assigned to the
    wrong person. Re-reading the issue after the race is the only way
    to catch that regression.
    """
    import threading
    client = mcp_client_with_open_bug
    results: list[dict] = []
    lock = threading.Lock()

    def attempt(who: str) -> None:
        r = client.call_tool("start_work", {
            "issue_id": client.bug_id, "assignee": who
        })
        with lock:
            results.append(json.loads(r.content[0].text))

    t1 = threading.Thread(target=attempt, args=("alice",))
    t2 = threading.Thread(target=attempt, args=("bob",))
    t1.start(); t2.start()
    t1.join(); t2.join()

    # Exactly one succeeded (returned IssueDict) and one CONFLICT.
    successes = [r for r in results if "assignee" in r]
    conflicts = [r for r in results if r.get("code") == ErrorCode.CONFLICT]
    assert len(successes) == 1
    assert len(conflicts) == 1

    # Re-read the issue from DB. Status must equal the canonical wip
    # (won the race → transition landed once) AND assignee must equal
    # the winner. If the transition had landed for both, we'd see the
    # loser's assignee persisted here too.
    final = client.call_tool("get_issue", {"issue_id": client.bug_id})
    final_payload = json.loads(final.content[0].text)
    winner = successes[0]["assignee"]
    assert final_payload["assignee"] == winner
    assert final_payload["status"] == "confirmed"
```

> to stress the DB-layer atomicity guarantee; don't rewrite it to async —
> that would hide the race the SQL is meant to prevent.

   ```python
   ) -> dict[str, Any]:

       Returns the full IssueDict on success.

       Raises:
           IssueNotFoundError: issue_id not in DB.
           ClaimConflictError: another assignee holds the claim.
   ```

   **Algorithm (single transaction, no nested commits):**

   ```python
       # Pre-flight read (read-only; safe outside the transaction).
       row = self.conn.execute(
           "SELECT type, status, assignee FROM issues WHERE id = ?",
           (issue_id,),
       ).fetchone()
       if row is None:
           raise IssueNotFoundError(issue_id)

       issue_type = row["type"]
       current_status = row["status"]
       current_assignee = row["assignee"] or ""

       # Idempotency: already-claimed-by-caller in wip → no-op return.
       # Resolve pack and state category for current_status first.
       # TemplateRegistry has NO `pack_for_type` method (verified
       # 2026-04-18, templates.py:245-555). The real API is:
       #   - `get_type(type_name) -> TypeTemplate | None` — templates.py:551
       #   - `get_pack(pack_name) -> WorkflowPack | None`  — templates.py:555
       # TypeTemplate carries its pack name on the `.pack` attribute
       # (templates.py:161).
       tpl = self.templates.get_type(issue_type)
       if tpl is None:
           raise InvalidTransitionError(issue_type, current_status)
       pack = self.templates.get_pack(tpl.pack)
       if pack is None:
           raise InvalidTransitionError(issue_type, current_status)
       state_by_name = {s.name: s for s in pack.types[issue_type].states}
       current_state = state_by_name.get(current_status)

       if current_state is not None and current_state.category == "wip" \
               and current_assignee == assignee:
           return self.get_issue(issue_id).to_dict()

       # If the issue is already in wip owned by a different assignee,
       # this is a CONFLICT (not INVALID_TRANSITION) — start_work is a
       # claim-and-transition composed op, so the claim guarantee
       # dominates.
       if current_state is not None and current_state.category == "wip" \
               and current_assignee and current_assignee != assignee:
           raise ClaimConflictError(issue_id, current_assignee=current_assignee)

       # Resolve transition target. target_status override takes priority.
       if target_status is not None:
           resolved_target = target_status
           # Validate: must be declared as a transition from current_status
           # and must be wip-category.
           valid = any(
               t.from_state == current_status and t.to_state == resolved_target
               for t in pack.types[issue_type].transitions
           )
           if not valid or state_by_name.get(resolved_target, None) is None \
                   or state_by_name[resolved_target].category != "wip":
               raise InvalidTransitionError(issue_type, current_status)
       else:
           # Propagates AmbiguousTransitionError / InvalidTransitionError.
           resolved_target = pack.canonical_working_status(
               issue_type, current_status=current_status
           )

       # ONE transaction: claim (CAS) + status transition + events.
       # Do NOT call claim_issue() here — it owns its own commit.
       # Do NOT use `with self.conn:` if sqlite3's context manager
       # semantics don't match the commit-on-success rule; the
       # explicit try/except mirrors the existing claim_issue pattern.
       try:
           claimed = self._claim_issue_nocommit(
               issue_id, assignee=assignee, actor=actor,
           )
           # Status transition — use the non-committing update path.
           # If `update_status` does not exist non-committingly, inline
           # the UPDATE and event record here (see existing db_issues
           # status-transition code for the SQL).
           self.conn.execute(
               "UPDATE issues SET status = ?, updated_at = ? WHERE id = ?",
               (resolved_target, _now_iso(), issue_id),
           )
           # Record the transition event (INSERT OR IGNORE to coexist
           # with the existing events dedup index — see the
           # events-table note in CLAUDE.md).
           self._record_event(
               issue_id, "status_changed",
               actor=actor or assignee,
               old_value=current_status,
               new_value=resolved_target,
           )
           self.conn.commit()
       except Exception:
           self.conn.rollback()
           raise

       return self.get_issue(issue_id).to_dict()
   ```

   **Why this shape:**
   - `_claim_issue_nocommit` (from Task 3.0) lets the claim and the
     transition share a single commit boundary. If the transition
     fails for any reason (SQLite constraint, transient I/O), the
     rollback reverts BOTH the assignee change and the status
     change — no half-state leaks.
   - We read `pack` and resolve the target before entering the
     write phase. The transition resolution is stable with respect
     to the row (current_status came from the same read), so racing
     writes between read and write are caught by the CAS inside
     `_claim_issue_nocommit`.
   - The idempotent "already claimed by me in wip" check fires before
     the transaction. It returns without touching the DB — two back-to-back
     `start_work` calls from the same agent produce one event, not two.

   **Events dedup:** The existing events table has a uniqueness index
   that produces a SQL error on duplicate `(issue_id, event_type, actor,
   old_value, new_value, created_at)` rows. Use `_record_event` which
   wraps the INSERT. If direct SQL is needed, prefer `INSERT OR IGNORE`
   to match the project convention (see CLAUDE.md memory on
   events-table dedup).

   **What NOT to do:**
   - Do NOT call `claim_issue()` (the public wrapper) here — it commits
     internally and breaks atomicity.
   - Do NOT call `update_issue(..., status=...)` if that method owns
     its own transaction; inline the UPDATE or add an internal
     non-committing variant first.
   - Do NOT rely on SQLite's implicit transaction — an explicit
     try/except/rollback mirrors the rest of `db_issues.py`.

```python
import json as json_mod
import sys

import click

from filigree.cli_commands._exits import cli_exit_for_code
from filigree.cli_common import get_db, refresh_summary
from filigree.types.api import (
    AmbiguousTransitionError,
    ClaimConflictError,
    ErrorCode,
    InvalidTransitionError,
    IssueNotFoundError,
)


@click.command("start-work")
@click.argument("issue_id")
@click.option("--assignee", required=True, help="Who is claiming (agent name).")
@click.option("--target-status", default=None, help="Explicit target status override.")
@click.option("--json", "as_json", is_flag=True, help="Output as JSON.")
@click.pass_context
def start_work(
    ctx: click.Context,
    issue_id: str,
    assignee: str,
    target_status: str | None,
    as_json: bool,
) -> None:
    """Atomically claim and advance an issue to its working status."""
    with get_db() as db:
        try:
            issue = db.start_work(
                issue_id=issue_id,
                assignee=assignee,
                target_status=target_status,
                actor=ctx.obj["actor"],
            )
        except IssueNotFoundError as e:
            _emit_cli_error(e, ErrorCode.NOT_FOUND, as_json=as_json)
            sys.exit(cli_exit_for_code(ErrorCode.NOT_FOUND))
        except ClaimConflictError as e:
            _emit_cli_error(
                e, ErrorCode.CONFLICT, as_json=as_json,
                details={"issue_id": e.issue_id, "current_assignee": e.current_assignee},
            )
            sys.exit(cli_exit_for_code(ErrorCode.CONFLICT))
        except InvalidTransitionError as e:
            _emit_cli_error(e, ErrorCode.INVALID_TRANSITION, as_json=as_json)
            sys.exit(cli_exit_for_code(ErrorCode.INVALID_TRANSITION))
        except AmbiguousTransitionError as e:
            _emit_cli_error(
                e, ErrorCode.VALIDATION, as_json=as_json,
                details={"candidates": e.candidates},
            )
            sys.exit(cli_exit_for_code(ErrorCode.VALIDATION))

        if as_json:
            click.echo(json_mod.dumps(issue, default=str))
        else:
            click.echo(
                f"Claimed {issue_id} → status={issue['status']} assignee={assignee}"
            )
        refresh_summary(db)


def _emit_cli_error(
    exc: Exception,
    code: ErrorCode,
    *,
    as_json: bool,
    details: dict | None = None,
) -> None:
    if as_json:
        payload = {"error": str(exc), "code": code.value}
        if details is not None:
            payload["details"] = details
        click.echo(json_mod.dumps(payload))
    else:
        click.echo(f"Error: {exc}", err=True)
```

```python
def test_start_next_work_concurrent_callers_claim_distinct_issues(
    mcp_client_with_ready_queue,
) -> None:
    """Two concurrent start_next_work calls with overlapping priority
    filters must claim distinct issues. Spec §3b explicit requirement.

    Without this guarantee, two agents doing the intuitive "grab
    highest-priority ready issue" pattern can both claim the same
    issue and one silently gets its transition reverted.
    """
    import threading

    client = mcp_client_with_ready_queue  # has P0, P1, P2 ready bugs
    results: list[dict] = []
    lock = threading.Lock()

    def attempt(who: str) -> None:
        r = client.call_tool("start_next_work", {"assignee": who})
        with lock:
            results.append(json.loads(r.content[0].text))

    t1 = threading.Thread(target=attempt, args=("alice",))
    t2 = threading.Thread(target=attempt, args=("bob",))
    t1.start(); t2.start()
    t1.join(); t2.join()

    successes = [r for r in results if "assignee" in r]
    # Both callers succeeded (different issues exist in the queue) OR
    # one succeeded and one got CONFLICT/NOT_FOUND (empty queue race).
    # The non-negotiable invariant: no two successes with the same id.
    success_ids = {s["id"] for s in successes}
    assert len(success_ids) == len(successes), (
        f"Two callers claimed the same issue: {results}"
    )
    # And the assignees must each match the caller that returned them
    # (can't have two successes with different assignees on the same row).
    for s in successes:
        assert s["assignee"] in ("alice", "bob")
```

Click 8.1+ supports aliases via a subclass. Add a helper to

```python
"""Click command class supporting permanent aliases."""
import click


class AliasedGroup(click.Group):
    """A Click group that allows multiple names per command."""

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._aliases: dict[str, str] = {}

    def add_command(self, cmd: click.Command, name: str | None = None,
                    aliases: list[str] | None = None) -> None:
        super().add_command(cmd, name)
        if aliases:
            canonical = name or cmd.name
            for alias in aliases:
                self._aliases[alias] = canonical

    def get_command(self, ctx: click.Context, cmd_name: str) -> click.Command | None:
        if cmd_name in self._aliases:
            cmd_name = self._aliases[cmd_name]
        return super().get_command(ctx, cmd_name)

    def list_commands(self, ctx: click.Context) -> list[str]:
        # Only canonical names in `--help` — aliases documented in command
        # docstrings.
        return sorted(self.commands.keys())
```

```markdown
  "newly_unblocked": [...]?}`. Legacy keys (`closed`, `errors`,
  `results`, `count`, `updated`, `unblocked`) removed.
- Batch operations accept `response_detail: "slim" | "full"` (default:
  `"slim"`).
- List operations return unified `ListResponse`: `{"items": [...],
  "has_more": bool, "next_offset": int?}`. `list_labels` renamed
  response key `labels` → `items`.
- Dashboard errors flattened from nested `{"error": {"message", "code"}}`
  to flat `{"error": str, "code": ErrorCode, "details": dict?}`.

### Breaking — error codes
- Closed 9-member `ErrorCode` enum replaces 15+ ad-hoc strings.
- `invalid`/`validation_error`/`invalid_path` → `VALIDATION`.
- `not_found`/`scanner_not_found`/`unknown_tool`/`command_not_found` →
  `NOT_FOUND`.
- `io_error`/`db_error`/`database_error`/`import_error` → `IO`.
- `permission_error` → `PERMISSION`, `not_initialized` → `NOT_INITIALIZED`,
  `stop_failed` → `STOP_FAILED`, `invalid_api_url` → `INVALID_API_URL`,
  `conflict` → `CONFLICT`, `invalid_transition` → `INVALID_TRANSITION`.

### Breaking — defaults
- `get_issue.include_files` defaults to `False` (was `True`). Opt in via
  `include_files=True` or CLI `--with-files`.
- CLI exit codes: `2` = validation, `3` = not initialized / schema
  mismatch, `4` = conflict (new), `1` = other operational errors.

### Added
- `start_work(issue_id, assignee, target_status?, actor?)` — atomic
  claim + workflow-pack-aware transition.
- `start_next_work(assignee, type?, priority_min?, priority_max?,
  target_status?, actor?)` — composed `claim_next + start_work`.
- CLI: `start-work`, `start-next-work`.
- Full CLI parity for observations (5), files/findings (8), scanners
  (6) — 19 new commands.
- MCP verb-noun aliases on existing CLI commands (permanent).
- `SchemaVersionMismatchError` — typed exception routed through `doctor`
  (exit 3), dashboard (clean exit), MCP server (warm-but-degraded mode).
- `filigree init` forward-warning + `.filigree/INSTALL_VERSION` marker.
- `claim_next` / `start_next_work` tie-break order documented.

### Removed
- All pre-2.0 response keys listed above in "Breaking — response
  envelope."
- All pre-2.0 error-code strings listed above in "Breaking — error
  codes."
```
