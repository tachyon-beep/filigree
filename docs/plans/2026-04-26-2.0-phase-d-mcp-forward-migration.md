# Phase D — MCP Forward-Migration to the Loom Living Shape

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate every MCP tool's input schema, handler logic, and response shape to the loom vocabulary (`issue_id` / `issue_ids`, `ListResponse[T]` envelopes, `BatchResponse[T]` envelopes, `state→status` rename) and add the composed `start_work` / `start_next_work` tools — so MCP reflects the living surface only, per ADR-002 §5.

**Architecture:** Phase D is a single coordinated rename + reshape across `src/filigree/mcp_tools/*` and `src/filigree/types/inputs.py`, with corresponding test churn in `tests/mcp/`. Classic HTTP stays frozen — MCP has no classic surface, so the rename does not touch `dashboard_routes/`. Each sub-task (D1–D7) is a logical commit boundary. The work is large (the work-package itself flags "probably several hundred test-line changes") but mechanically uniform.

**Tech Stack:** Python 3.13, MCP Python SDK (`mcp.types.Tool`), JSON Schema, pytest-asyncio. No new dependencies. The `WorkflowPack.canonical_working_status` helper added in D6 uses existing `AmbiguousTransitionError` and `InvalidTransitionError` from `src/filigree/types/api.py:504/524`.

## Operational rules

- **Single commit boundary per sub-task** (D1, D2, ... D7) — seven commits, not one. Sub-tasks ship in dependency order so each commit's tests pass on their own.
- **Pre-push checklist before every commit:** `uv run ruff check src/ tests/`, `uv run ruff format --check src/ tests/`, `uv run mypy src/filigree/`, `uv run pytest --tb=short`. All four must be green.
- **Do not push, switch branches, or create branches without explicit user approval.** Branch is `2.0-project-management-extension`.
- **Classic stays frozen.** No edits to `dashboard_routes/issues.py::create_classic_router()` or any other classic-router factory. MCP has no classic — the vocabulary shift only affects MCP and its TypedDicts.
- **Wide test churn is expected.** The handover prompt warns "expect wide test churn — probably several hundred test-line changes" in `tests/mcp/test_tools.py`. Do not bargain that down by leaving back-compat shims; the whole point of Phase D is removing them.
- **No legacy compat shims.** When renaming an inputSchema field from `id` to `issue_id`, do not also accept `id` as a fallback. The MCP surface is forward-only — clients re-pin against the new schema. The HTTP loom generation is the stable target for federation consumers.

## Design decisions (locked in before tasks)

1. **MCP is forward-only.** No legacy compat shims accepting both `id` and `issue_id`. Documented prominently in the Phase D commit messages.
2. **Per-entity primary key naming.** Issue PK fields → `issue_id`/`issue_ids`. Observation PK fields → `observation_id`/`observation_ids`. Finding PK fields → `finding_id`/`finding_ids`. Comment PK fields → `comment_id`. Event PK fields → `event_id`. Cross-entity references keep their existing names (`parent_id` becomes `parent_issue_id` per the work-package, but `blocked_by`, `blocks`, `children` arrays of issue ids stay unchanged because they're collection names not field references — verify with a grep before D3 commit).
3. **`get_issue.include_files` defaults to `False`.** D4 flip is a behavioral change. MCP tests that expected `files` in responses without explicitly opting in must change to pass `include_files=True` or assert files-absent.
4. **`state` → `status` rename is global on MCP and CLI.** `get_workflow_states` becomes `get_workflow_statuses`; `explain_state` becomes `explain_status`. CLI command names follow. The internal column in `issues.status` is already named `status` (it always was), so no DB migration is needed — the rename is only at the MCP/CLI surface.
5. **`start_work` is atomic.** Implemented as a savepoint wrapping `claim_issue` + `update_issue` to a target status. If the workflow pack defines exactly one status in the `wip` category (the common case), `target_status` is optional and defaults to that. Otherwise `target_status` is required, and an `AmbiguousTransitionError` surfaces with the candidate set if the caller omitted it on a multi-wip pack.
6. **`start_next_work` is `claim_next` + atomic transition.** Same atomicity pattern. Uses the same priority/created_at/issue_id ordering documented in `db.claim_next`.
7. **`SlimIssue.id` → `SlimIssue.issue_id` is the most disruptive single change.** Classic HTTP must continue returning `id` on the wire because classic-batch-update currently returns `{updated: [IssueDict]}` and search returns `{results: [IssueDict]}` — these are full IssueDicts using `IssueDict.id`, NOT `SlimIssue`. So `SlimIssue.id` rename touches MCP only. Verify with a grep: `SlimIssue` should appear only in MCP code paths and `BatchResponse[SlimIssue]` types. Classic HTTP returns `IssueDict`, which keeps `id`. (See D3 verification step.)

**D2 design call — does MCP `list_observations` keep its `stats` sibling?**

The loom HTTP `/api/loom/observations` (Phase C4) dropped `stats` per the strict envelope rule. MCP has historically returned `{observations, stats, has_more}`. Document the deprecation in the commit message.

- [ ] **Step 1: Add `_list_response` helper in `mcp_tools/common.py`**

Add this function alongside `_apply_has_more` (which becomes a private helper used by `_list_response`):

```python
def _list_response(items: list[Any], *, has_more: bool, next_offset: int | None = None) -> dict[str, Any]:
    """Build a unified ListResponse[T] envelope for MCP list tools.

    Mirrors filigree.generations.loom.adapters.list_response — same
    rule: next_offset is present only when has_more is True. Defined
    here as a free helper rather than reusing the loom adapter to keep
    the MCP surface free of generation-layer dependencies (per the
    operating principle 'MCP reflects the living surface only',
    not 'MCP imports loom').
    """
    body: dict[str, Any] = {"items": items, "has_more": has_more}
    if has_more and next_offset is not None:
        body["next_offset"] = next_offset
    return body
```

```python
def canonical_working_status(self) -> str:
    """Return the unique wip-category status name for this type.

    Used by start_work / start_next_work to default ``target_status``
    when the pack defines exactly one wip status. Raises
```

```python
    if the type has multiple wip statuses (caller must specify).
    """
    with self.conn:  # savepoint via context manager
        self.conn.execute("SAVEPOINT start_work")
        try:
            issue = self.claim_issue(issue_id, assignee=assignee, actor=actor)
            if target_status is None:
                template = self.templates.get_type_template(issue.type)
                if template is None:
                    self.conn.execute("ROLLBACK TO SAVEPOINT start_work")
                    raise InvalidTransitionError(f"Unknown type {issue.type!r}")
                target_status = template.canonical_working_status()
            updated = self.update_issue(issue_id, status=target_status, actor=actor)
        except Exception:
            self.conn.execute("ROLLBACK TO SAVEPOINT start_work")
            raise
        else:
            self.conn.execute("RELEASE SAVEPOINT start_work")
            return updated
```

```python
def start_next_work(
    self,
    *,
    assignee: str,
    type: str | None = None,
    priority_min: int | None = None,
    priority_max: int | None = None,
    target_status: str | None = None,
    actor: str = "",

    if provided) and atomically transition it to a working status.
    Returns None if no ready issue matches the filters.
    """

        self.conn.execute("SAVEPOINT start_next_work")
        try:

                assignee=assignee,
                actor=actor,

                priority_min=priority_min,
                priority_max=priority_max,
            )

                if template is None:
                    raise InvalidTransitionError(f"Unknown type {issue.type!r}")
                target_status = template.canonical_working_status()
            updated = self.update_issue(issue.id, status=target_status, actor=actor)
        except Exception:

            raise
        else:
            self.conn.execute("RELEASE SAVEPOINT start_next_work")
```
